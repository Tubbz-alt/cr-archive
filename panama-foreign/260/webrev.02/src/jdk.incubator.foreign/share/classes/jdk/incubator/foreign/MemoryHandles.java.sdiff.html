 41 /**
 42  * This class defines several factory methods for constructing and combining memory access var handles.
 43  * To obtain a memory access var handle, clients must start from one of the <em>leaf</em> methods
 44  * (see {@link MemoryHandles#varHandle(Class, ByteOrder)},
 45  * {@link MemoryHandles#varHandle(Class, long, ByteOrder)}). This determines the variable type
 46  * (all primitive types but {@code void} and {@code boolean} are supported), as well as the alignment constraint and the
 47  * byte order associated to a memory access var handle. The resulting memory access var handle can then be combined in various ways
 48  * to emulate different addressing modes. The var handles created by this class feature a <em>mandatory</em> coordinate type
 49  * (of type {@link MemoryAddress}), and zero or more {@code long} coordinate types, which can be used to emulate
 50  * multi-dimensional array indexing.
 51  * <p>
 52  * As an example, consider the memory layout expressed by a {@link SequenceLayout} instance constructed as follows:
 53  * <blockquote><pre>{@code
 54 SequenceLayout seq = MemoryLayout.ofSequence(5,
 55     MemoryLayout.ofStruct(
 56         MemoryLayout.ofPaddingBits(32),
 57         MemoryLayout.ofValueBits(32, ByteOrder.BIG_ENDIAN).withName("value")
 58     ));
 59  * }</pre></blockquote>
 60  * To access the member layout named {@code value}, we can construct a memory access var handle as follows:
 61  * <blockquote><pre>{@code
 62 VarHandle handle = MemoryHandles.varHandle(int.class, ByteOrder.BIG_ENDIAN); //(MemoryAddress) -> int
 63 handle = MemoryHandles.withOffset(handle, 4); //(MemoryAddress) -> int
 64 handle = MemoryHandles.withStride(handle, 8); //(MemoryAddress, long) -> int
 65  * }</pre></blockquote>
 66  *
 67  * <h2>Addressing mode</h2>
 68  *
 69  * The final memory location accessed by a memory access var handle can be computed as follows:
 70  *
 71  * <blockquote><pre>{@code
 72 address = base + offset
 73  * }</pre></blockquote>
 74  *
 75  * where {@code base} denotes the address expressed by the {@link MemoryAddress} access coordinate, and {@code offset}
 76  * can be expressed in the following form:
 77  *
 78  * <blockquote><pre>{@code
 79 offset = c_1 + c_2 + ... + c_m + (x_1 * s_1) + (x_2 * s_2) + ... + (x_n * s_n)
 80  * }</pre></blockquote>
 81  *
 82  * where {@code x_1}, {@code x_2}, ... {@code x_n} are <em>dynamic</em> values provided as optional {@code long}
 83  * access coordinates, whereas {@code c_1}, {@code c_2}, ... {@code c_m} and {@code s_0}, {@code s_1}, ... {@code s_n} are
 84  * <em>static</em> constants which are can be acquired through the {@link MemoryHandles#withOffset(VarHandle, long)}
 85  * and the {@link MemoryHandles#withStride(VarHandle, long)} combinators, respectively.
 86  *
 87  * <h2><a id="memaccess-mode"></a>Alignment and access modes</h2>
 88  *
 89  * A memory access var handle is associated with an access size {@code S} and an alignment constraint {@code B}
 90  * (both expressed in bytes). We say that a memory access operation is <em>fully aligned</em> if it occurs
 91  * at a memory address {@code A} which is compatible with both alignment constraints {@code S} and {@code B}.
 92  * If access is fully aligned then following access modes are supported and are
 93  * guaranteed to support atomic access:
 94  * <ul>
 95  * <li>read write access modes for all {@code T}, with the exception of
 96  *     access modes {@code get} and {@code set} for {@code long} and
 97  *     {@code double} on 32-bit platforms.
 98  * <li>atomic update access modes for {@code int}, {@code long},
 99  *     {@code float} or {@code double}.
100  *     (Future major platform releases of the JDK may support additional
101  *     types for certain currently unsupported access modes.)
102  * <li>numeric atomic update access modes for {@code int} and {@code long}.
103  *     (Future major platform releases of the JDK may support additional
104  *     numeric types for certain currently unsupported access modes.)
105  * <li>bitwise atomic update access modes for {@code int} and {@code long}.
106  *     (Future major platform releases of the JDK may support additional

113  * {@link Double#doubleToRawLongBits}, respectively).
114  * <p>
115  * Alternatively, a memory access operation is <em>partially aligned</em> if it occurs at a memory address {@code A}
116  * which is only compatible with the alignment constraint {@code B}; in such cases, access for anything other than the
117  * {@code get} and {@code set} access modes will result in an {@code IllegalStateException}. If access is partially aligned,
118  * atomic access is only guaranteed with respect to the largest power of two that divides the GCD of {@code A} and {@code S}.
119  * <p>
120  * Finally, in all other cases, we say that a memory access operation is <em>misaligned</em>; in such cases an
121  * {@code IllegalStateException} is thrown, irrespective of the access mode being used.
122  */

238     /**
239      * Returns a var handle that adds a <em>fixed</em> offset to the incoming {@link MemoryAddress} coordinate
240      * and then propagates such value to the target var handle. That is,
241      * when the returned var handle receives a memory address coordinate pointing at a memory location at
242      * offset <em>O</em>, a memory address coordinate pointing at a memory location at offset <em>O' + O</em>
243      * is created, and then passed to the target var handle.
244      *
245      * The returned var handle will feature the same type and access coordinates as the target var handle.
246      *
247      * @param target the target memory access handle to access after the offset adjustment.
248      * @param bytesOffset the offset, in bytes. Must be positive or zero.
249      * @return the adapted var handle.
250      * @throws IllegalArgumentException if the first access coordinate type is not of type {@link MemoryAddress}.
251      */
252     public static VarHandle withOffset(VarHandle target, long bytesOffset) {
253         if (bytesOffset == 0) {
254             return target; //nothing to do
255         }
256 
257         checkAddressFirstCoordinate(target);
258 
259         if (JLI.isMemoryAccessVarHandle(target) &&
260                 (bytesOffset & JLI.memoryAddressAlignmentMask(target)) == 0) {
261             //flatten
262             return Utils.fixUpVarHandle(JLI.memoryAccessVarHandle(
263                     JLI.memoryAddressCarrier(target),
264                     JLI.memoryAddressAlignmentMask(target),
265                     JLI.memoryAddressByteOrder(target),
266                     JLI.memoryAddressOffset(target) + bytesOffset,
267                     JLI.memoryAddressStrides(target)));
268         } else {
269             //slow path
270             VarHandle res = collectCoordinates(target, 0, ADD_OFFSET);
271             return insertCoordinates(res, 1, bytesOffset);
272         }
273     }

275     /**
276      * Returns a var handle which adds a <em>variable</em> offset to the incoming {@link MemoryAddress}
277      * access coordinate value and then propagates such value to the target var handle.
278      * That is, when the returned var handle receives a memory address coordinate pointing at a memory location at
279      * offset <em>O</em>, a new memory address coordinate pointing at a memory location at offset <em>(S * X) + O</em>
280      * is created, and then passed to the target var handle,
281      * where <em>S</em> is a constant <em>stride</em>, whereas <em>X</em> is a dynamic value that will be
282      * provided as an additional access coordinate (of type {@code long}).
283      *
284      * The returned var handle will feature the same type as the target var handle; an additional access coordinate
285      * of type {@code long} will be added to the access coordinate types of the target var handle at the position
286      * immediately following the leading access coordinate of type {@link MemoryAddress}.
287      *
288      * @param target the target memory access handle to access after the scale adjustment.
289      * @param bytesStride the stride, in bytes, by which to multiply the coordinate value.
290      * @return the adapted var handle.
291      * @throws IllegalArgumentException if the first access coordinate type is not of type {@link MemoryAddress}.
292      */
293     public static VarHandle withStride(VarHandle target, long bytesStride) {
294         if (bytesStride == 0) {
295             return dropCoordinates(target, 1, long.class); // dummy coordinate
296         }
297 
298         checkAddressFirstCoordinate(target);
299 
300         if (JLI.isMemoryAccessVarHandle(target) &&
301                 (bytesStride & JLI.memoryAddressAlignmentMask(target)) == 0) {
302             //flatten
303             long[] strides = JLI.memoryAddressStrides(target);
304             long[] newStrides = new long[strides.length + 1];
305             System.arraycopy(strides, 0, newStrides, 1, strides.length);
306             newStrides[0] = bytesStride;
307 
308             return Utils.fixUpVarHandle(JLI.memoryAccessVarHandle(
309                     JLI.memoryAddressCarrier(target),
310                     JLI.memoryAddressAlignmentMask(target),
311                     JLI.memoryAddressByteOrder(target),
312                     JLI.memoryAddressOffset(target),
313                     newStrides));
314         } else {
315             //slow path
316             VarHandle res = collectCoordinates(target, 0, ADD_STRIDE);
317             return insertCoordinates(res, 2, bytesStride);
318         }
319     }

321     /**
322      * Adapt an existing var handle into a new var handle whose carrier type is {@link MemoryAddress}.
323      * That is, when calling {@link VarHandle#get(Object...)} on the returned var handle,
324      * the read numeric value will be turned into a memory address (as if by calling {@link MemoryAddress#ofLong(long)});
325      * similarly, when calling {@link VarHandle#set(Object...)}, the memory address to be set will be converted
326      * into a numeric value, and then written into memory. The amount of bytes read (resp. written) from (resp. to)
327      * memory depends on the carrier of the original memory access var handle.
328      *
329      * @param target the memory access var handle to be adapted
330      * @return the adapted var handle.
331      * @throws IllegalArgumentException if the carrier type of {@code varHandle} is either {@code boolean},
332      * {@code float}, or {@code double}, or is not a primitive type.
333      */
334     public static VarHandle asAddressVarHandle(VarHandle target) {
335         Class<?> carrier = target.varType();
336         if (!carrier.isPrimitive() || carrier == boolean.class ||
337                 carrier == float.class || carrier == double.class) {
338             throw new IllegalArgumentException("Unsupported carrier type: " + carrier.getName());
339         }
340 
341         if (carrier != long.class) {
342             // slow-path, we need to adapt

345                     MethodHandles.explicitCastArguments(LONG_TO_ADDRESS, MethodType.methodType(MemoryAddress.class, carrier)));
346         } else {
347             // fast-path
348             return filterValue(target, ADDRESS_TO_LONG, LONG_TO_ADDRESS);
349         }
350     }

353      * Adapts a target var handle by narrowing incoming values and widening
354      * outgoing values, to and from the given type, respectively.
355      * <p>
356      * The returned var handle can be used to conveniently treat unsigned
357      * primitive data types as if they were a wider signed primitive type. For
358      * example, it is often convenient to model an <i>unsigned short</i> as a
359      * Java {@code int} to avoid dealing with negative values, which would be
360      * the case if modeled as a Java {@code short}. This is illustrated in the following example:
361      * <blockquote><pre>{@code
362     MemorySegment segment = MemorySegment.allocateNative(2);
363     VarHandle SHORT_VH = MemoryLayouts.JAVA_SHORT.varHandle(short.class);
364     VarHandle INT_VH = MemoryHandles.asUnsigned(SHORT_VH, int.class);
365     SHORT_VH.set(segment.address(), (short)-1);
366     INT_VH.get(segment.address()); // returns 65535
367      * }</pre></blockquote>
368      * <p>
369      * When calling e.g. {@link VarHandle#set(Object...)} on the resulting var
370      * handle, the incoming value (of type {@code adaptedType}) is converted by a
371      * <i>narrowing primitive conversion</i> and then passed to the {@code
372      * target} var handle. A narrowing primitive conversion may lose information
373      * about the overall magnitude of a numeric value. Conversely, when calling
374      * e.g. {@link VarHandle#get(Object...)} on the resulting var handle, the
375      * returned value obtained from the {@code target} var handle is converted
376      * by a <i>unsigned widening conversion</i> before being returned to the
377      * caller. In an unsigned widening conversion the high-order bits greater
378      * than that of the {@code target} carrier type are zero, and the low-order
379      * bits (equal to the width of the {@code target} carrier type) are equal to
380      * the bits of the value obtained from the {@code target} var handle.
381      * <p>
382      * The returned var handle will feature the variable type {@code adaptedType},
383      * and the same access coordinates, the same access modes (see {@link
384      * java.lang.invoke.VarHandle.AccessMode}, and the same atomic access
385      * guarantees, as those featured by the {@code target} var handle.
386      *

603      * target var handle's coordinate types. If {@code pos} is zero, the dummy coordinates will precede
604      * the target's real arguments; if {@code pos} is <i>N</i> they will come after.
605      * <p>
606      * The resulting var handle will feature the same access modes (see {@link java.lang.invoke.VarHandle.AccessMode}) and
607      * atomic access guarantees as those featured by the target var handle.

183      *
184      * The accessed offset is zero.

194      * @throws IllegalArgumentException if an illegal carrier type is used, or if {@code alignmentBytes} is not a power of two.

200                 || (alignmentBytes & (alignmentBytes - 1)) != 0) { // is power of 2?

202         }
203

209      * That is, when calling {@link VarHandle#get(Object...)} on the returned var handle,
210      * the read numeric value will be turned into a memory address (as if by calling {@link MemoryAddress#ofLong(long)});
211      * similarly, when calling {@link VarHandle#set(Object...)}, the memory address to be set will be converted
212      * into a numeric value, and then written into memory. The amount of bytes read (resp. written) from (resp. to)

214      *

222         if (!carrier.isPrimitive() || carrier == boolean.class ||
223                 carrier == float.class || carrier == double.class) {
224             throw new IllegalArgumentException("Unsupported carrier type: " + carrier.getName());
225         }
226 
227         if (carrier != long.class) {
228             // slow-path, we need to adapt

231                     MethodHandles.explicitCastArguments(LONG_TO_ADDRESS, MethodType.methodType(MemoryAddress.class, carrier)));
232         } else {
233             // fast-path
234             return filterValue(target, ADDRESS_TO_LONG, LONG_TO_ADDRESS);
235         }

240      * outgoing values, to and from the given type, respectively.
241      * <p>
242      * The returned var handle can be used to conveniently treat unsigned
243      * primitive data types as if they were a wider signed primitive type. For
244      * example, it is often convenient to model an <i>unsigned short</i> as a
245      * Java {@code int} to avoid dealing with negative values, which would be
246      * the case if modeled as a Java {@code short}. This is illustrated in the following example:
247      * <blockquote><pre>{@code
248     MemorySegment segment = MemorySegment.allocateNative(2);
249     VarHandle SHORT_VH = MemoryLayouts.JAVA_SHORT.varHandle(short.class);
250     VarHandle INT_VH = MemoryHandles.asUnsigned(SHORT_VH, int.class);
251     SHORT_VH.set(segment, (short)-1);
252     INT_VH.get(segment); // returns 65535
253      * }</pre></blockquote>
254      * <p>
255      * When calling e.g. {@link VarHandle#set(Object...)} on the resulting var
256      * handle, the incoming value (of type {@code adaptedType}) is converted by a
257      * <i>narrowing primitive conversion</i> and then passed to the {@code
258      * target} var handle. A narrowing primitive conversion may lose information
259      * about the overall magnitude of a numeric value. Conversely, when calling
260      * e.g. {@link VarHandle#get(Object...)} on the resulting var handle, the
261      * returned value obtained from the {@code target} var handle is converted
262      * by a <i>unsigned widening conversion</i> before being returned to the
263      * caller. In an unsigned widening conversion the high-order bits greater
264      * than that of the {@code target} carrier type are zero, and the low-order
265      * bits (equal to the width of the {@code target} carrier type) are equal to
266      * the bits of the value obtained from the {@code target} var handle.
267      * <p>
268      * The returned var handle will feature the variable type {@code adaptedType},
269      * and the same access coordinates, the same access modes (see {@link
270      * java.lang.invoke.VarHandle.AccessMode}, and the same atomic access
271      * guarantees, as those featured by the {@code target} var handle.
272      *

489      * target var handle's coordinate types. If {@code pos} is zero, the dummy coordinates will precede
490      * the target's real arguments; if {@code pos} is <i>N</i> they will come after.
491      * <p>
492      * The resulting var handle will feature the same access modes (see {@link java.lang.invoke.VarHandle.AccessMode}) and
493      * atomic access guarantees as those featured by the target var handle.
494      *
495      * @param target the var handle to invoke after the dummy coordinates are dropped
496      * @param pos position of first coordinate to drop (zero for the leftmost)
