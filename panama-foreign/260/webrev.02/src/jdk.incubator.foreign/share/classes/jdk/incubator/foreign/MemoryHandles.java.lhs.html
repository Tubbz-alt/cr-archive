 41 /**
 42  * This class defines several factory methods for constructing and combining memory access var handles.
 43  * To obtain a memory access var handle, clients must start from one of the <em>leaf</em> methods
 44  * (see {@link MemoryHandles#varHandle(Class, ByteOrder)},
 45  * {@link MemoryHandles#varHandle(Class, long, ByteOrder)}). This determines the variable type
 46  * (all primitive types but {@code void} and {@code boolean} are supported), as well as the alignment constraint and the
 47  * byte order associated to a memory access var handle. The resulting memory access var handle can then be combined in various ways
 48  * to emulate different addressing modes. The var handles created by this class feature a <em>mandatory</em> coordinate type
 49  * (of type {@link MemoryAddress}), and zero or more {@code long} coordinate types, which can be used to emulate
 50  * multi-dimensional array indexing.
 51  * <p>
 52  * As an example, consider the memory layout expressed by a {@link SequenceLayout} instance constructed as follows:
 53  * <blockquote><pre>{@code
 54 SequenceLayout seq = MemoryLayout.ofSequence(5,
 55     MemoryLayout.ofStruct(
 56         MemoryLayout.ofPaddingBits(32),
 57         MemoryLayout.ofValueBits(32, ByteOrder.BIG_ENDIAN).withName("value")
 58     ));
 59  * }</pre></blockquote>
 60  * To access the member layout named {@code value}, we can construct a memory access var handle as follows:
 61  * <blockquote><pre>{@code
 62 VarHandle handle = MemoryHandles.varHandle(int.class, ByteOrder.BIG_ENDIAN); //(MemoryAddress) -> int
 63 handle = MemoryHandles.withOffset(handle, 4); //(MemoryAddress) -> int
 64 handle = MemoryHandles.withStride(handle, 8); //(MemoryAddress, long) -> int
 65  * }</pre></blockquote>
 66  *
 67  * <h2>Addressing mode</h2>
 68  *
 69  * The final memory location accessed by a memory access var handle can be computed as follows:
 70  *
 71  * <blockquote><pre>{@code
 72 address = base + offset
 73  * }</pre></blockquote>
 74  *
 75  * where {@code base} denotes the address expressed by the {@link MemoryAddress} access coordinate, and {@code offset}
 76  * can be expressed in the following form:
 77  *
 78  * <blockquote><pre>{@code
 79 offset = c_1 + c_2 + ... + c_m + (x_1 * s_1) + (x_2 * s_2) + ... + (x_n * s_n)
 80  * }</pre></blockquote>
 81  *
 82  * where {@code x_1}, {@code x_2}, ... {@code x_n} are <em>dynamic</em> values provided as optional {@code long}
 83  * access coordinates, whereas {@code c_1}, {@code c_2}, ... {@code c_m} and {@code s_0}, {@code s_1}, ... {@code s_n} are
 84  * <em>static</em> constants which are can be acquired through the {@link MemoryHandles#withOffset(VarHandle, long)}
 85  * and the {@link MemoryHandles#withStride(VarHandle, long)} combinators, respectively.
 86  *
 87  * <h2><a id="memaccess-mode"></a>Alignment and access modes</h2>
 88  *
 89  * A memory access var handle is associated with an access size {@code S} and an alignment constraint {@code B}
 90  * (both expressed in bytes). We say that a memory access operation is <em>fully aligned</em> if it occurs
 91  * at a memory address {@code A} which is compatible with both alignment constraints {@code S} and {@code B}.
 92  * If access is fully aligned then following access modes are supported and are
 93  * guaranteed to support atomic access:
 94  * <ul>
 95  * <li>read write access modes for all {@code T}, with the exception of
 96  *     access modes {@code get} and {@code set} for {@code long} and
 97  *     {@code double} on 32-bit platforms.
 98  * <li>atomic update access modes for {@code int}, {@code long},
 99  *     {@code float} or {@code double}.
100  *     (Future major platform releases of the JDK may support additional
101  *     types for certain currently unsupported access modes.)
102  * <li>numeric atomic update access modes for {@code int} and {@code long}.
103  *     (Future major platform releases of the JDK may support additional
104  *     numeric types for certain currently unsupported access modes.)
105  * <li>bitwise atomic update access modes for {@code int} and {@code long}.
106  *     (Future major platform releases of the JDK may support additional
107  *     numeric types for certain currently unsupported access modes.)
108  * </ul>
109  *
110  * If {@code T} is {@code float} or {@code double} then atomic
111  * update access modes compare values using their bitwise representation
112  * (see {@link Float#floatToRawIntBits} and
113  * {@link Double#doubleToRawLongBits}, respectively).
114  * <p>
115  * Alternatively, a memory access operation is <em>partially aligned</em> if it occurs at a memory address {@code A}
116  * which is only compatible with the alignment constraint {@code B}; in such cases, access for anything other than the
117  * {@code get} and {@code set} access modes will result in an {@code IllegalStateException}. If access is partially aligned,
118  * atomic access is only guaranteed with respect to the largest power of two that divides the GCD of {@code A} and {@code S}.
119  * <p>
120  * Finally, in all other cases, we say that a memory access operation is <em>misaligned</em>; in such cases an
121  * {@code IllegalStateException} is thrown, irrespective of the access mode being used.
122  */

184     /**
185      * Creates a memory access var handle with the given carrier type and byte order.
186      *
187      * The resulting memory access var handle features a single {@link MemoryAddress} access coordinate,
188      * and its variable type is set by the given carrier type.
189      *
190      * The alignment constraint for the resulting memory access var handle is the same as the in memory size of the
191      * carrier type, and the accessed offset is set at zero.
192      *
193      * @apiNote the resulting var handle features certain <a href="#memaccess-mode">access mode restrictions</a>,
194      * which are common to all memory access var handles.
195      *
196      * @param carrier the carrier type. Valid carriers are {@code byte}, {@code short}, {@code char}, {@code int},
197      * {@code float}, {@code long}, and {@code double}.
198      * @param byteOrder the required byte order.
199      * @return the new memory access var handle.
200      * @throws IllegalArgumentException when an illegal carrier type is used
201      */
202     public static VarHandle varHandle(Class<?> carrier, ByteOrder byteOrder) {
203         checkCarrier(carrier);
204         return varHandle(carrier,
205                 carrierSize(carrier),
206                 byteOrder);
207     }

209     /**
210      * Creates a memory access var handle with the given carrier type, alignment constraint, and byte order.
211      *
212      * The resulting memory access var handle features a single {@link MemoryAddress} access coordinate,
213      * and its variable type is set by the given carrier type.
214      *
215      * The accessed offset is zero.
216      *
217      * @apiNote the resulting var handle features certain <a href="#memaccess-mode">access mode restrictions</a>,
218      * which are common to all memory access var handles.
219      *
220      * @param carrier the carrier type. Valid carriers are {@code byte}, {@code short}, {@code char}, {@code int},
221      * {@code float}, {@code long}, and {@code double}.
222      * @param alignmentBytes the alignment constraint (in bytes). Must be a power of two.
223      * @param byteOrder the required byte order.
224      * @return the new memory access var handle.
225      * @throws IllegalArgumentException if an illegal carrier type is used, or if {@code alignmentBytes} is not a power of two.
226      */
227     public static VarHandle varHandle(Class<?> carrier, long alignmentBytes, ByteOrder byteOrder) {
228         checkCarrier(carrier);
229 
230         if (alignmentBytes <= 0
231                 || (alignmentBytes & (alignmentBytes - 1)) != 0) { // is power of 2?
232             throw new IllegalArgumentException("Bad alignment: " + alignmentBytes);
233         }
234 
235         return Utils.fixUpVarHandle(JLI.memoryAccessVarHandle(carrier, alignmentBytes - 1, byteOrder, 0, new long[]{}));
236     }

238     /**
239      * Returns a var handle that adds a <em>fixed</em> offset to the incoming {@link MemoryAddress} coordinate
240      * and then propagates such value to the target var handle. That is,
241      * when the returned var handle receives a memory address coordinate pointing at a memory location at
242      * offset <em>O</em>, a memory address coordinate pointing at a memory location at offset <em>O' + O</em>
243      * is created, and then passed to the target var handle.
244      *
245      * The returned var handle will feature the same type and access coordinates as the target var handle.
246      *
247      * @param target the target memory access handle to access after the offset adjustment.
248      * @param bytesOffset the offset, in bytes. Must be positive or zero.
249      * @return the adapted var handle.
250      * @throws IllegalArgumentException if the first access coordinate type is not of type {@link MemoryAddress}.
251      */
252     public static VarHandle withOffset(VarHandle target, long bytesOffset) {
253         if (bytesOffset == 0) {
254             return target; //nothing to do
255         }
256 
257         checkAddressFirstCoordinate(target);
258 
259         if (JLI.isMemoryAccessVarHandle(target) &&
260                 (bytesOffset & JLI.memoryAddressAlignmentMask(target)) == 0) {
261             //flatten
262             return Utils.fixUpVarHandle(JLI.memoryAccessVarHandle(
263                     JLI.memoryAddressCarrier(target),
264                     JLI.memoryAddressAlignmentMask(target),
265                     JLI.memoryAddressByteOrder(target),
266                     JLI.memoryAddressOffset(target) + bytesOffset,
267                     JLI.memoryAddressStrides(target)));
268         } else {
269             //slow path
270             VarHandle res = collectCoordinates(target, 0, ADD_OFFSET);
271             return insertCoordinates(res, 1, bytesOffset);
272         }
273     }

275     /**
276      * Returns a var handle which adds a <em>variable</em> offset to the incoming {@link MemoryAddress}
277      * access coordinate value and then propagates such value to the target var handle.
278      * That is, when the returned var handle receives a memory address coordinate pointing at a memory location at
279      * offset <em>O</em>, a new memory address coordinate pointing at a memory location at offset <em>(S * X) + O</em>
280      * is created, and then passed to the target var handle,
281      * where <em>S</em> is a constant <em>stride</em>, whereas <em>X</em> is a dynamic value that will be
282      * provided as an additional access coordinate (of type {@code long}).
283      *
284      * The returned var handle will feature the same type as the target var handle; an additional access coordinate
285      * of type {@code long} will be added to the access coordinate types of the target var handle at the position
286      * immediately following the leading access coordinate of type {@link MemoryAddress}.
287      *
288      * @param target the target memory access handle to access after the scale adjustment.
289      * @param bytesStride the stride, in bytes, by which to multiply the coordinate value.
290      * @return the adapted var handle.
291      * @throws IllegalArgumentException if the first access coordinate type is not of type {@link MemoryAddress}.
292      */
293     public static VarHandle withStride(VarHandle target, long bytesStride) {
294         if (bytesStride == 0) {
295             return dropCoordinates(target, 1, long.class); // dummy coordinate
296         }
297 
298         checkAddressFirstCoordinate(target);
299 
300         if (JLI.isMemoryAccessVarHandle(target) &&
301                 (bytesStride & JLI.memoryAddressAlignmentMask(target)) == 0) {
302             //flatten
303             long[] strides = JLI.memoryAddressStrides(target);
304             long[] newStrides = new long[strides.length + 1];
305             System.arraycopy(strides, 0, newStrides, 1, strides.length);
306             newStrides[0] = bytesStride;
307 
308             return Utils.fixUpVarHandle(JLI.memoryAccessVarHandle(
309                     JLI.memoryAddressCarrier(target),
310                     JLI.memoryAddressAlignmentMask(target),
311                     JLI.memoryAddressByteOrder(target),
312                     JLI.memoryAddressOffset(target),
313                     newStrides));
314         } else {
315             //slow path
316             VarHandle res = collectCoordinates(target, 0, ADD_STRIDE);
317             return insertCoordinates(res, 2, bytesStride);
318         }
319     }

321     /**
322      * Adapt an existing var handle into a new var handle whose carrier type is {@link MemoryAddress}.
323      * That is, when calling {@link VarHandle#get(Object...)} on the returned var handle,
324      * the read numeric value will be turned into a memory address (as if by calling {@link MemoryAddress#ofLong(long)});
325      * similarly, when calling {@link VarHandle#set(Object...)}, the memory address to be set will be converted
326      * into a numeric value, and then written into memory. The amount of bytes read (resp. written) from (resp. to)
327      * memory depends on the carrier of the original memory access var handle.
328      *
329      * @param target the memory access var handle to be adapted
330      * @return the adapted var handle.
331      * @throws IllegalArgumentException if the carrier type of {@code varHandle} is either {@code boolean},
332      * {@code float}, or {@code double}, or is not a primitive type.
333      */
334     public static VarHandle asAddressVarHandle(VarHandle target) {
335         Class<?> carrier = target.varType();
336         if (!carrier.isPrimitive() || carrier == boolean.class ||
337                 carrier == float.class || carrier == double.class) {
338             throw new IllegalArgumentException("Unsupported carrier type: " + carrier.getName());
339         }
340 
341         if (carrier != long.class) {
342             // slow-path, we need to adapt
343             return filterValue(target,
344                     MethodHandles.explicitCastArguments(ADDRESS_TO_LONG, MethodType.methodType(carrier, MemoryAddress.class)),
345                     MethodHandles.explicitCastArguments(LONG_TO_ADDRESS, MethodType.methodType(MemoryAddress.class, carrier)));
346         } else {
347             // fast-path
348             return filterValue(target, ADDRESS_TO_LONG, LONG_TO_ADDRESS);
349         }
350     }

352     /**
353      * Adapts a target var handle by narrowing incoming values and widening
354      * outgoing values, to and from the given type, respectively.
355      * <p>
356      * The returned var handle can be used to conveniently treat unsigned
357      * primitive data types as if they were a wider signed primitive type. For
358      * example, it is often convenient to model an <i>unsigned short</i> as a
359      * Java {@code int} to avoid dealing with negative values, which would be
360      * the case if modeled as a Java {@code short}. This is illustrated in the following example:
361      * <blockquote><pre>{@code
362     MemorySegment segment = MemorySegment.allocateNative(2);
363     VarHandle SHORT_VH = MemoryLayouts.JAVA_SHORT.varHandle(short.class);
364     VarHandle INT_VH = MemoryHandles.asUnsigned(SHORT_VH, int.class);
365     SHORT_VH.set(segment.address(), (short)-1);
366     INT_VH.get(segment.address()); // returns 65535
367      * }</pre></blockquote>
368      * <p>
369      * When calling e.g. {@link VarHandle#set(Object...)} on the resulting var
370      * handle, the incoming value (of type {@code adaptedType}) is converted by a
371      * <i>narrowing primitive conversion</i> and then passed to the {@code
372      * target} var handle. A narrowing primitive conversion may lose information
373      * about the overall magnitude of a numeric value. Conversely, when calling
374      * e.g. {@link VarHandle#get(Object...)} on the resulting var handle, the
375      * returned value obtained from the {@code target} var handle is converted
376      * by a <i>unsigned widening conversion</i> before being returned to the
377      * caller. In an unsigned widening conversion the high-order bits greater
378      * than that of the {@code target} carrier type are zero, and the low-order
379      * bits (equal to the width of the {@code target} carrier type) are equal to
380      * the bits of the value obtained from the {@code target} var handle.
381      * <p>
382      * The returned var handle will feature the variable type {@code adaptedType},
383      * and the same access coordinates, the same access modes (see {@link
384      * java.lang.invoke.VarHandle.AccessMode}, and the same atomic access
385      * guarantees, as those featured by the {@code target} var handle.
386      *
387      * @param target the memory access var handle to be adapted
388      * @param adaptedType the adapted type
389      * @return the adapted var handle.
390      * @throws IllegalArgumentException if the carrier type of {@code target}
391      * is not one of {@code byte}, {@code short}, or {@code int}; if {@code
392      * adaptedType} is not one of {@code int}, or {@code long}; if the bitwidth
393      * of the {@code adaptedType} is not greater than that of the {@code target}
394      * carrier type
395      * @throws NullPointerException if either of {@code target} or {@code
396      * adaptedType} is null
397      *
398      * @jls 5.1.3 Narrowing Primitive Conversion
399      */
400     public static VarHandle asUnsigned(VarHandle target, final Class<?> adaptedType) {
401         Objects.requireNonNull(target);
402         Objects.requireNonNull(adaptedType);
403         final Class<?> carrier = target.varType();
404         checkWidenable(carrier);
405         checkNarrowable(adaptedType);
406         checkTargetWiderThanCarrier(carrier, adaptedType);
407 
408         if (adaptedType == int.class && carrier == byte.class) {
409             return filterValue(target, INT_TO_BYTE, BYTE_TO_UNSIGNED_INT);
410         } else if (adaptedType == int.class && carrier == short.class) {
411             return filterValue(target, INT_TO_SHORT, SHORT_TO_UNSIGNED_INT);
412         } else if (adaptedType == long.class && carrier == byte.class) {
413             return filterValue(target, LONG_TO_BYTE, BYTE_TO_UNSIGNED_LONG);
414         } else if (adaptedType == long.class && carrier == short.class) {
415             return filterValue(target, LONG_TO_SHORT, SHORT_TO_UNSIGNED_LONG);
416         } else if (adaptedType == long.class && carrier == int.class) {
417             return filterValue(target, LONG_TO_INT, INT_TO_UNSIGNED_LONG);
418         } else {
419             throw new InternalError("should not reach here");
420         }
421     }

423     /**
424      * Adapts a target var handle by pre-processing incoming and outgoing values using a pair of filter functions.
425      * <p>
426      * When calling e.g. {@link VarHandle#set(Object...)} on the resulting var handle, the incoming value (of type {@code T}, where
427      * {@code T} is the <em>last</em> parameter type of the first filter function) is processed using the first filter and then passed
428      * to the target var handle.
429      * Conversely, when calling e.g. {@link VarHandle#get(Object...)} on the resulting var handle, the return value obtained from
430      * the target var handle (of type {@code T}, where {@code T} is the <em>last</em> parameter type of the second filter function)
431      * is processed using the second filter and returned to the caller. More advanced access mode types, such as
432      * {@link java.lang.invoke.VarHandle.AccessMode#COMPARE_AND_EXCHANGE} might apply both filters at the same time.
433      * <p>
434      * For the boxing and unboxing filters to be well formed, their types must be of the form {@code (A... , S) -> T} and
435      * {@code (A... , T) -> S}, respectively, where {@code T} is the type of the target var handle. If this is the case,
436      * the resulting var handle will have type {@code S} and will feature the additional coordinates {@code A...} (which
437      * will be appended to the coordinates of the target var handle).
438      * <p>
439      * The resulting var handle will feature the same access modes (see {@link java.lang.invoke.VarHandle.AccessMode} and
440      * atomic access guarantees as those featured by the target var handle.
441      *
442      * @param target the target var handle
443      * @param filterToTarget a filter to convert some type {@code S} into the type of {@code target}
444      * @param filterFromTarget a filter to convert the type of {@code target} to some type {@code S}
445      * @return an adapter var handle which accepts a new type, performing the provided boxing/unboxing conversions.
446      * @throws NullPointerException if either {@code target}, {@code filterToTarget} or {@code filterFromTarget} are {@code == null}.
447      * @throws IllegalArgumentException if {@code filterFromTarget} and {@code filterToTarget} are not well-formed, that is, they have types
448      * other than {@code (A... , S) -> T} and {@code (A... , T) -> S}, respectively, where {@code T} is the type of the target var handle,
449      * or if either {@code filterFromTarget} or {@code filterToTarget} throws any checked exceptions.
450      */
451     public static VarHandle filterValue(VarHandle target, MethodHandle filterToTarget, MethodHandle filterFromTarget) {
452         return JLI.filterValue(target, filterToTarget, filterFromTarget);
453     }

455     /**
456      * Adapts a target var handle by pre-processing incoming coordinate values using unary filter functions.
457      * <p>
458      * When calling e.g. {@link VarHandle#get(Object...)} on the resulting var handle, the incoming coordinate values
459      * starting at position {@code pos} (of type {@code C1, C2 ... Cn}, where {@code C1, C2 ... Cn} are the return type
460      * of the unary filter functions) are transformed into new values (of type {@code S1, S2 ... Sn}, where {@code S1, S2 ... Sn} are the
461      * parameter types of the unary filter functions), and then passed (along with any coordinate that was left unaltered
462      * by the adaptation) to the target var handle.
463      * <p>
464      * For the coordinate filters to be well formed, their types must be of the form {@code S1 -> T1, S2 -> T1 ... Sn -> Tn},
465      * where {@code T1, T2 ... Tn} are the coordinate types starting at position {@code pos} of the target var handle.
466      * <p>
467      * The resulting var handle will feature the same access modes (see {@link java.lang.invoke.VarHandle.AccessMode}) and
468      * atomic access guarantees as those featured by the target var handle.
469      *
470      * @param target the target var handle
471      * @param pos the position of the first coordinate to be transformed
472      * @param filters the unary functions which are used to transform coordinates starting at position {@code pos}
473      * @return an adapter var handle which accepts new coordinate types, applying the provided transformation
474      * to the new coordinate values.
475      * @throws NullPointerException if either {@code target}, {@code filters} are {@code == null}.
476      * @throws IllegalArgumentException if the handles in {@code filters} are not well-formed, that is, they have types
477      * other than {@code S1 -> T1, S2 -> T2, ... Sn -> Tn} where {@code T1, T2 ... Tn} are the coordinate types starting
478      * at position {@code pos} of the target var handle, if {@code pos} is not between 0 and the target var handle coordinate arity, inclusive,

498      * atomic access guarantees as those featured by the target var handle.
499      *
500      * @param target the var handle to invoke after the bound coordinates are inserted
501      * @param pos the position of the first coordinate to be inserted
502      * @param values the series of bound coordinates to insert
503      * @return an adapter var handle which inserts an additional coordinates,
504      *         before calling the target var handle
505      * @throws NullPointerException if either {@code target}, {@code values} are {@code == null}.
506      * @throws IllegalArgumentException if {@code pos} is not between 0 and the target var handle coordinate arity, inclusive,
507      * or if more values are provided than the actual number of coordinate types available starting at {@code pos}.
508      * @throws ClassCastException if the bound coordinates in {@code values} are not well-formed, that is, they have types
509      * other than {@code T1, T2 ... Tn }, where {@code T1, T2 ... Tn} are the coordinate types starting at position {@code pos}
510      * of the target var handle.
511      */

516     /**
517      * Provides a var handle which adapts the coordinate values of the target var handle, by re-arranging them
518      * so that the new coordinates match the provided ones.
519      * <p>
520      * The given array controls the reordering.
521      * Call {@code #I} the number of incoming coordinates (the value
522      * {@code newCoordinates.size()}, and call {@code #O} the number
523      * of outgoing coordinates (the number of coordinates associated with the target var handle).
524      * Then the length of the reordering array must be {@code #O},
525      * and each element must be a non-negative number less than {@code #I}.
526      * For every {@code N} less than {@code #O}, the {@code N}-th
527      * outgoing coordinate will be taken from the {@code I}-th incoming
528      * coordinate, where {@code I} is {@code reorder[N]}.
529      * <p>
530      * No coordinate value conversions are applied.
531      * The type of each incoming coordinate, as determined by {@code newCoordinates},
532      * must be identical to the type of the corresponding outgoing coordinate
533      * in the target var handle.
534      * <p>
535      * The reordering array need not specify an actual permutation.
536      * An incoming coordinate will be duplicated if its index appears
537      * more than once in the array, and an incoming coordinate will be dropped
538      * if its index does not appear in the array.
539      * <p>
540      * The resulting var handle will feature the same access modes (see {@link java.lang.invoke.VarHandle.AccessMode}) and
541      * atomic access guarantees as those featured by the target var handle.
542      * @param target the var handle to invoke after the coordinates have been reordered
543      * @param newCoordinates the new coordinate types
544      * @param reorder an index array which controls the reordering
545      * @return an adapter var handle which re-arranges the incoming coordinate values,
546      * before calling the target var handle
547      * @throws NullPointerException if either {@code target}, {@code newCoordinates} or {@code reorder} are {@code == null}.
548      * @throws IllegalArgumentException if the index array length is not equal to
549      * the number of coordinates of the target var handle, or if any index array element is not a valid index for
550      * a coordinate of {@code newCoordinates}, or if two corresponding coordinate types in
551      * the target var handle and in {@code newCoordinates} are not identical.
552      */
553     public static VarHandle permuteCoordinates(VarHandle target, List<Class<?>> newCoordinates, int... reorder) {
554         return JLI.permuteCoordinates(target, newCoordinates, reorder);
555     }

563      * <p>

565      * type {@code R} as its coordinate in position {@code pos}, preceded and/or followed by

568      * replaces (in order) the whole subsequence of coordinates originally
569      * passed to the adapter.
570      * <p>
571      * The argument types (if any) of the filter
572      * replace zero or one coordinate types of the target var handle, at position {@code pos},
573      * in the resulting adapted var handle.
574      * The return type of the filter must be identical to the
575      * coordinate type of the target var handle at position {@code pos}, and that target var handle
576      * coordinate is supplied by the return value of the filter.
577      * <p>
578      * The resulting var handle will feature the same access modes (see {@link java.lang.invoke.VarHandle.AccessMode}) and
579      * atomic access guarantees as those featured by the target var handle.

581      * @param target the var handle to invoke after the coordinates have been filtered
582      * @param pos the position of the coordinate to be filtered
583      * @param filter the filter method handle
584      * @return an adapter var handle which filters the incoming coordinate values,
585      * before calling the target var handle
586      * @throws NullPointerException if either {@code target}, {@code filter} are {@code == null}.
587      * @throws IllegalArgumentException if the return type of {@code filter}
588      * is void, or it is not the same as the {@code pos} coordinate of the target var handle,
589      * if {@code pos} is not between 0 and the target var handle coordinate arity, inclusive,
590      * if the resulting var handle's type would have <a href="MethodHandle.html#maxarity">too many coordinates</a>,
