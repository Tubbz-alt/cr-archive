 44 /**
 45  * A memory segment models a contiguous region of memory. A memory segment is associated with both spatial
 46  * and temporal bounds. Spatial bounds ensure that memory access operations on a memory segment cannot affect a memory location
 47  * which falls <em>outside</em> the boundaries of the memory segment being accessed. Temporal bounds ensure that memory access
 48  * operations on a segment cannot occur after a memory segment has been closed (see {@link MemorySegment#close()}).
 49  * <p>
 50  * All implementations of this interface must be <a href="{@docRoot}/java.base/java/lang/doc-files/ValueBased.html">value-based</a>;
 51  * use of identity-sensitive operations (including reference equality ({@code ==}), identity hash code, or synchronization) on
 52  * instances of {@code MemorySegment} may have unpredictable results and should be avoided. The {@code equals} method should
 53  * be used for comparisons.
 54  * <p>
 55  * Non-platform classes should not implement {@linkplain MemorySegment} directly.
 56  *
 57  * <h2>Constructing memory segments from different sources</h2>
 58  *
 59  * There are multiple ways to obtain a memory segment. First, memory segments backed by off-heap memory can
 60  * be allocated using one of the many factory methods provided (see {@link MemorySegment#allocateNative(MemoryLayout)},
 61  * {@link MemorySegment#allocateNative(long)} and {@link MemorySegment#allocateNative(long, long)}). Memory segments obtained
 62  * in this way are called <em>native memory segments</em>.
 63  * <p>
 64  * It is also possible to obtain a memory segment backed by an existing heap-allocated Java array,
 65  * using one of the provided factory methods (e.g. {@link MemorySegment#ofArray(int[])}). Memory segments obtained
 66  * in this way are called <em>array memory segments</em>.
 67  * <p>
 68  * It is possible to obtain a memory segment backed by an existing Java byte buffer (see {@link ByteBuffer}),
 69  * using the factory method {@link MemorySegment#ofByteBuffer(ByteBuffer)}.
 70  * Memory segments obtained in this way are called <em>buffer memory segments</em>. Note that buffer memory segments might
 71  * be backed by native memory (as in the case of native memory segments) or heap memory (as in the case of array memory segments),
 72  * depending on the characteristics of the byte buffer instance the segment is associated with. For instance, a buffer memory
 73  * segment obtained from a byte buffer created with the {@link ByteBuffer#allocateDirect(int)} method will be backed
 74  * by native memory.
 75  * <p>
 76  * Finally, it is also possible to obtain a memory segment backed by a memory-mapped file using the factory method
 77  * {@link MemorySegment#mapFromPath(Path, long, long, FileChannel.MapMode)}. Such memory segments are called <em>mapped memory segments</em>
 78  * (see {@link MappedMemorySegment}).
 79  * <p>
 80  * Array and buffer segments are effectively <em>views</em> over existing memory regions which might outlive the
 81  * lifecycle of the segments derived from them, and can even be manipulated directly (e.g. via array access, or direct use
 82  * of the {@link ByteBuffer} API) by other clients. As a result, while sharing array or buffer segments is possible,
 83  * it is strongly advised that clients wishing to do so take extra precautions to make sure that the underlying memory sources
 84  * associated with such segments remain inaccessible, and that said memory sources are never aliased by more than one segment
 85  * at a time - e.g. so as to prevent concurrent modifications of the contents of an array, or buffer segment.
 86  *
 87  * <h2>Closing a memory segment</h2>
 88  *
 89  * Memory segments are closed explicitly (see {@link MemorySegment#close()}). When a segment is closed, it is no longer
 90  * <em>alive</em> (see {@link #isAlive()}, and subsequent operation on the segment (or on any {@link MemoryAddress} instance
 91  * derived from it) will fail with {@link IllegalStateException}.
 92  * <p>
 93  * Closing a segment might trigger the releasing of the underlying memory resources associated with said segment, depending on
 94  * the kind of memory segment being considered:
 95  * <ul>
 96  *     <li>closing a native memory segment results in <em>freeing</em> the native memory associated with it</li>
 97  *     <li>closing a mapped memory segment results in the backing memory-mapped file to be unmapped</li>
 98  *     <li>closing a buffer, or a heap segment does not have any side-effect, other than marking the segment
 99  *     as <em>not alive</em> (see {@link MemorySegment#isAlive()}). Also, since the buffer and heap segments might keep
100  *     strong references to the original buffer or array instance, it is the responsibility of clients to ensure that
101  *     these segments are discarded in a timely manner, so as not to prevent garbage collection to reclaim the underlying
102  *     objects.</li>
103  * </ul>
104  *
105  * <h2><a id = "access-modes">Access modes</a></h2>
106  *
107  * Memory segments supports zero or more <em>access modes</em>. Supported access modes are {@link #READ},
108  * {@link #WRITE}, {@link #CLOSE}, {@link #ACQUIRE} and {@link #HANDOFF}. The set of access modes supported by a segment alters the
109  * set of operations that are supported by that segment. For instance, attempting to call {@link #close()} on
110  * a segment which does not support the {@link #CLOSE} access mode will result in an exception.
111  * <p>
112  * The set of supported access modes can only be made stricter (by supporting <em>fewer</em> access modes). This means
113  * that restricting the set of access modes supported by a segment before sharing it with other clients
114  * is generally a good practice if the creator of the segment wants to retain some control over how the segment
115  * is going to be accessed.
116  *
117  * <h2>Memory segment views</h2>
118  *
119  * Memory segments support <em>views</em>. For instance, it is possible to alter the set of supported access modes,
120  * by creating an <em>immutable</em> view of a memory segment, as follows:
121  * <blockquote><pre>{@code
122 MemorySegment segment = ...
123 MemorySegment roSegment = segment.withAccessModes(segment.accessModes() & ~WRITE);
124  * }</pre></blockquote>
125  * It is also possible to create views whose spatial bounds are stricter than the ones of the original segment
126  * (see {@link MemorySegment#asSlice(long, long)}).
127  * <p>
128  * Temporal bounds of the original segment are inherited by the view; that is, closing a segment view, such as a sliced
129  * view, will cause the original segment to be closed; as such special care must be taken when sharing views
130  * between multiple clients. If a client want to protect itself against early closure of a segment by
131  * another actor, it is the responsibility of that client to take protective measures, such as removing {@link #CLOSE}
132  * from the set of supported access modes, before sharing the view with another client.
133  * <p>
134  * To allow for interoperability with existing code, a byte buffer view can be obtained from a memory segment
135  * (see {@link #asByteBuffer()}). This can be useful, for instance, for those clients that want to keep using the
136  * {@link ByteBuffer} API, but need to operate on large memory segments. Byte buffers obtained in such a way support
137  * the same spatial and temporal access restrictions associated to the memory segment from which they originated.
138  *
139  * <h2><a id = "thread-confinement">Thread confinement</a></h2>
140  *
141  * Memory segments support strong thread-confinement guarantees. Upon creation, they are assigned an <em>owner thread</em>,
142  * typically the thread which initiated the creation operation. After creation, only the owner thread will be allowed
143  * to directly manipulate the memory segment (e.g. close the memory segment) or access the underlying memory associated with
144  * the segment using a memory access var handle. Any attempt to perform such operations from a thread other than the
145  * owner thread will result in a runtime failure.
146  * <p>
147  * Memory segments support <em>serial thread confinement</em>; that is, ownership of a memory segment can change (see
148  * {@link #withOwnerThread(Thread)}). This allows, for instance, for two threads {@code A} and {@code B} to share
149  * a segment in a controlled, cooperative and race-free fashion.
150  * <p>
151  * In some cases, it might be useful for multiple threads to process the contents of the same memory segment concurrently
152  * (e.g. in the case of parallel processing); while memory segments provide strong confinement guarantees, it is possible
153  * to obtain a {@link Spliterator} from a segment, which can be used to slice the segment and allow multiple thread to
154  * work in parallel on disjoint segment slices (this assumes that the access mode {@link #ACQUIRE} is set).
155  * For instance, the following code can be used to sum all int values in a memory segment in parallel:
156  * <blockquote><pre>{@code
157 MemorySegment segment = ...
158 SequenceLayout SEQUENCE_LAYOUT = MemoryLayout.ofSequence(1024, MemoryLayouts.JAVA_INT);
159 VarHandle VH_int = SEQUENCE_LAYOUT.elementLayout().varHandle(int.class);
160 int sum = StreamSupport.stream(MemorySegment.spliterator(segment, SEQUENCE_LAYOUT), true)
161                        .mapToInt(s -> (int)VH_int.get(s.address()))
162                        .sum();
163  * }</pre></blockquote>
164  *
165  * @apiNote In the future, if the Java language permits, {@link MemorySegment}
166  * may become a {@code sealed} interface, which would prohibit subclassing except by
167  * {@link MappedMemorySegment} and other explicitly permitted subtypes.
168  *
169  * @implSpec
170  * Implementations of this interface are immutable, thread-safe and <a href="{@docRoot}/java.base/java/lang/doc-files/ValueBased.html">value-based</a>.
171  */
172 public interface MemorySegment extends Addressable, AutoCloseable {

174     /**
175      * The base memory address associated with this memory segment. The returned address is
176      * a <em>checked</em> memory address and can therefore be used in dereference operations
177      * (see {@link MemoryAddress}).
178      * @return The base memory address.
179      * @throws IllegalStateException if this segment is not <em>alive</em>, or if access occurs from a thread other than the
180      * thread owning this segment

185     /**
186      * Returns a spliterator for the given memory segment. The returned spliterator reports {@link Spliterator#SIZED},
187      * {@link Spliterator#SUBSIZED}, {@link Spliterator#IMMUTABLE}, {@link Spliterator#NONNULL} and {@link Spliterator#ORDERED}
188      * characteristics.
189      * <p>
190      * The returned spliterator splits the segment according to the specified sequence layout; that is,
191      * if the supplied layout is a sequence layout whose element count is {@code N}, then calling {@link Spliterator#trySplit()}
192      * will result in a spliterator serving approximatively {@code N/2} elements (depending on whether N is even or not).
193      * As such, splitting is possible as long as {@code N >= 2}. The spliterator returns segments that feature the same
194      * <a href="#access-modes">access modes</a> as the given segment less the {@link #CLOSE} access mode.
195      * <p>
196      * The returned spliterator effectively allows to slice a segment into disjoint sub-segments, which can then
197      * be processed in parallel by multiple threads (if the access mode {@link #ACQUIRE} is set).
198      * While closing the segment (see {@link #close()}) during pending concurrent execution will generally
199      * fail with an exception, it is possible to close a segment when a spliterator has been obtained but no thread
200      * is actively working on it using {@link Spliterator#tryAdvance(Consumer)}; in such cases, any subsequent call
201      * to {@link Spliterator#tryAdvance(Consumer)} will fail with an exception.
202      * @param segment the segment to be used for splitting.
203      * @param layout the layout to be used for splitting.
204      * @param <S> the memory segment type
205      * @return the element spliterator for this segment
206      * @throws IllegalStateException if the segment is not <em>alive</em>, or if access occurs from a thread other than the
207      * thread owning this segment
208      */
209     static <S extends MemorySegment> Spliterator<S> spliterator(S segment, SequenceLayout layout) {
210         return AbstractMemorySegmentImpl.spliterator(segment, layout);
211     }

227      *

234      * @throws NullPointerException if {@code newOwner == null}
235      * @throws IllegalArgumentException if the segment is already a confined segment owner by {@code newOnwer}.
236      * @throws UnsupportedOperationException if this segment does not support the {@link #HANDOFF} access mode.
237      */

247      * Obtains a segment view with specific <a href="#access-modes">access modes</a>. Supported access modes are {@link #READ}, {@link #WRITE},

323     /**
324      * Fills a value into this memory segment.
325      * <p>
326      * More specifically, the given value is filled into each address of this
327      * segment. Equivalent to (but likely more efficient than) the following code:
328      *
329      * <pre>{@code
330 byteHandle = MemoryLayout.ofSequence(MemoryLayouts.JAVA_BYTE)
331          .varHandle(byte.class, MemoryLayout.PathElement.sequenceElement());
332 for (long l = 0; l < segment.byteSize(); l++) {
333      byteHandle.set(segment.address(), l, value);
334 }
335      * }</pre>
336      *
337      * without any regard or guarantees on the ordering of particular memory
338      * elements being set.
339      * <p>
340      * Fill can be useful to initialize or reset the memory of a segment.
341      *
342      * @param value the value to fill into this segment
343      * @return this memory segment
344      * @throws IllegalStateException if this segment is not <em>alive</em>, or if access occurs from a thread other than the
345      * thread owning this segment
346      * @throws UnsupportedOperationException if this segment does not support the {@link #WRITE} access mode
347      */
348     MemorySegment fill(byte value);

350     /**
351      * Performs a bulk copy from given source segment to this segment. More specifically, the bytes at
352      * offset {@code 0} through {@code src.byteSize() - 1} in the source segment are copied into this segment
353      * at offset {@code 0} through {@code src.byteSize() - 1}.
354      * If the source segment overlaps with this segment, then the copying is performed as if the bytes at
355      * offset {@code 0} through {@code src.byteSize() - 1} in the source segment were first copied into a
356      * temporary segment with size {@code bytes}, and then the contents of the temporary segment were copied into
357      * this segment at offset {@code 0} through {@code src.byteSize() - 1}.
358      * <p>
359      * The result of a bulk copy is unspecified if, in the uncommon case, the source segment and this segment
360      * do not overlap, but refer to overlapping regions of the same backing storage using different addresses.
361      * For example, this may occur if the same file is {@link MemorySegment#mapFromPath mapped} to two segments.
362      *
363      * @param src the source segment.
364      * @throws IndexOutOfBoundsException if {@code src.byteSize() > this.byteSize()}.
365      * @throws IllegalStateException if either the source segment or this segment have been already closed,
366      * or if access occurs from a thread other than the thread owning either segment.
367      * @throws UnsupportedOperationException if either the source segment or this segment do not feature required access modes;
368      * more specifically, {@code src} should feature at least the {@link MemorySegment#READ} access mode,
369      * while this segment should feature at least the {@link MemorySegment#WRITE} access mode.
370      */
371     void copyFrom(MemorySegment src);

373     /**
374      * Finds and returns the offset, in bytes, of the first mismatch between
375      * this segment and a given other segment. The offset is relative to the
376      * {@link #address() base address} of each segment and will be in the
377      * range of 0 (inclusive) up to the {@link #byteSize() size} (in bytes) of
378      * the smaller memory segment (exclusive).
379      * <p>
380      * If the two segments share a common prefix then the returned offset is
381      * the length of the common prefix and it follows that there is a mismatch
382      * between the two segments at that offset within the respective segments.
383      * If one segment is a proper prefix of the other then the returned offset is
384      * the smaller of the segment sizes, and it follows that the offset is only
385      * valid for the larger segment. Otherwise, there is no mismatch and {@code
386      * -1} is returned.
387      *
388      * @param other the segment to be tested for a mismatch with this segment
389      * @return the relative offset, in bytes, of the first mismatch between this
390      * and the given other segment, otherwise -1 if no mismatch
391      * @throws IllegalStateException if either this segment of the other segment
392      * have been already closed, or if access occurs from a thread other than the
393      * thread owning either segment
394      * @throws UnsupportedOperationException if either this segment or the other
395      * segment does not feature at least the {@link MemorySegment#READ} access mode
396      */
397     long mismatch(MemorySegment other);

399     /**
400      * Wraps this segment in a {@link ByteBuffer}. Some of the properties of the returned buffer are linked to
401      * the properties of this segment. For instance, if this segment is <em>immutable</em>
402      * (e.g. the segment has access mode {@link #READ} but not {@link #WRITE}), then the resulting buffer is <em>read-only</em>
403      * (see {@link ByteBuffer#isReadOnly()}. Additionally, if this is a native memory segment, the resulting buffer is
404      * <em>direct</em> (see {@link ByteBuffer#isDirect()}).
405      * <p>
406      * The life-cycle of the returned buffer will be tied to that of this segment. That means that if the this segment
407      * is closed (see {@link MemorySegment#close()}, accessing the returned
408      * buffer will throw an {@link IllegalStateException}.
409      * <p>
410      * The resulting buffer's byte order is {@link java.nio.ByteOrder#BIG_ENDIAN}; this can be changed using
411      * {@link ByteBuffer#order(java.nio.ByteOrder)}.
412      *
413      * @return a {@link ByteBuffer} view of this memory segment.
414      * @throws UnsupportedOperationException if this segment cannot be mapped onto a {@link ByteBuffer} instance,
415      * e.g. because it models an heap-based segment that is not based on a {@code byte[]}), or if its size is greater
416      * than {@link Integer#MAX_VALUE}, or if the segment does not support the {@link #READ} access mode.
417      */
418     ByteBuffer asByteBuffer();

425      * @throws IllegalStateException if this segment has been closed, or if access occurs from a thread other than the

678      * Creates a new native memory segment that models a newly allocated block of off-heap memory with given size and
679      * alignment constraint (in bytes). The segment will feature all <a href="#access-modes">access modes</a>
680      * (see {@link #ALL_ACCESS}).
681      *
682      * @implNote The block of off-heap memory associated with the returned native memory segment is initialized to zero.
683      * Moreover, a client is responsible to call the {@link MemorySegment#close()} on a native memory segment,
684      * to make sure the backing off-heap memory block is deallocated accordingly. Failure to do so will result in off-heap memory leaks.
685      *
686      * @param bytesSize the size (in bytes) of the off-heap memory block backing the native memory segment.
687      * @param alignmentBytes the alignment constraint (in bytes) of the off-heap memory block backing the native memory segment.
688      * @return a new native memory segment.
689      * @throws IllegalArgumentException if {@code bytesSize < 0}, {@code alignmentBytes < 0}, or if {@code alignmentBytes}
690      * is not a power of 2.
691      */
692     static MemorySegment allocateNative(long bytesSize, long alignmentBytes) {
693         if (bytesSize <= 0) {
694             throw new IllegalArgumentException("Invalid allocation size : " + bytesSize);
695         }
696 
697         if (alignmentBytes < 0 ||
698                 ((alignmentBytes & (alignmentBytes - 1)) != 0L)) {
699             throw new IllegalArgumentException("Invalid alignment constraint : " + alignmentBytes);
700         }
701 
702         return NativeMemorySegmentImpl.makeNativeSegment(bytesSize, alignmentBytes);
703     }

729      * Returns a new native memory segment with given base address and size; the returned segment has its own temporal
730      * bounds, and can therefore be closed; closing such a segment can optionally result in calling an user-provided cleanup
731      * action. This method can be very useful when interacting with custom native memory sources (e.g. custom allocators,
732      * GPU memory, etc.), where an address to some underlying memory region is typically obtained from native code
733      * (often as a plain {@code long} value). The segment will feature all <a href="#access-modes">access modes</a>
734      * (see {@link #ALL_ACCESS}).
735      * <p>
736      * This method is <em>restricted</em>. Restricted methods are unsafe, and, if used incorrectly, their use might crash
737      * the JVM or, worse, silently result in memory corruption. Thus, clients should refrain from depending on
738      * restricted methods, and use safe and supported functionalities, where possible.
739      *
740      * @param addr the desired base address
741      * @param bytesSize the desired size.
742      * @param owner the desired owner thread. If {@code owner == null}, the returned segment is <em>not</em> confined.
743      * @param cleanup a cleanup action to be executed when the {@link MemorySegment#close()} method is called on the
744      *                returned segment. If {@code cleanup == null}, no cleanup action is executed.
745      * @param attachment an object that must be kept alive by the returned segment; this can be useful when
746      *                   the returned segment depends on memory which could be released if a certain object
747      *                   is determined to be unreacheable. In most cases this will be set to {@code null}.
748      * @return a new native memory segment with given base address, size, owner, cleanup action and object attachment.
749      * @throws IllegalArgumentException if {@code bytesSize <= 0}.
750      * @throws UnsupportedOperationException if {@code addr} is associated with an heap segment.

754      */

758             throw new IllegalArgumentException("Invalid size : " + bytesSize);
759         }
760         Utils.checkRestrictedAccess("MemorySegment.ofNativeRestricted");
761         return NativeMemorySegmentImpl.makeNativeSegmentUnchecked(addr, bytesSize, owner, cleanup, attachment);
