 39 /**
 40  * A memory segment models a contiguous region of memory. A memory segment is associated with both spatial
 41  * and temporal bounds. Spatial bounds ensure that memory access operations on a memory segment cannot affect a memory location
 42  * which falls <em>outside</em> the boundaries of the memory segment being accessed. Temporal checks ensure that memory access
 43  * operations on a segment cannot occur after a memory segment has been closed (see {@link MemorySegment#close()}).
 44  * <p>
 45  * All implementations of this interface must be <a href="{@docRoot}/java.base/java/lang/doc-files/ValueBased.html">value-based</a>;
 46  * use of identity-sensitive operations (including reference equality ({@code ==}), identity hash code, or synchronization) on
 47  * instances of {@code MemorySegment} may have unpredictable results and should be avoided. The {@code equals} method should
 48  * be used for comparisons.
 49  * <p>
 50  * Non-platform classes should not implement {@linkplain MemorySegment} directly.
 51  *
 52  * <h2>Constructing memory segments from different sources</h2>
 53  *
 54  * There are multiple ways to obtain a memory segment. First, memory segments backed by off-heap memory can
 55  * be allocated using one of the many factory methods provided (see {@link MemorySegment#allocateNative(MemoryLayout)},
 56  * {@link MemorySegment#allocateNative(long)} and {@link MemorySegment#allocateNative(long, long)}). Memory segments obtained
 57  * in this way are called <em>native memory segments</em>.
 58  * <p>
 59  * It is also possible to obtain a memory segment backed by an existing heap-allocated Java array,
 60  * using one of the provided factory methods (e.g. {@link MemorySegment#ofArray(int[])}). Memory segments obtained
 61  * in this way are called <em>array memory segments</em>.
 62  * <p>
 63  * It is possible to obtain a memory segment backed by an existing Java byte buffer (see {@link ByteBuffer}),
 64  * using the factory method {@link MemorySegment#ofByteBuffer(ByteBuffer)}.
 65  * Memory segments obtained in this way are called <em>buffer memory segments</em>. Note that buffer memory segments might
 66  * be backed by native memory (as in the case of native memory segments) or heap memory (as in the case of array memory segments),
 67  * depending on the characteristics of the byte buffer instance the segment is associated with. For instance, a buffer memory
 68  * segment obtained from a byte buffer created with the {@link ByteBuffer#allocateDirect(int)} method will be backed
 69  * by native memory.
 70  * <p>
 71  * Finally, it is also possible to obtain a memory segment backed by a memory-mapped file using the factory method
 72  * {@link MemorySegment#mapFromPath(Path, long, FileChannel.MapMode)}. Such memory segments are called <em>mapped memory segments</em>.
 73  *
 74  * <h2>Closing a memory segment</h2>
 75  *
 76  * Memory segments are closed explicitly (see {@link MemorySegment#close()}). In general when a segment is closed, all off-heap
 77  * resources associated with it are released; this has different meanings depending on the kind of memory segment being
 78  * considered:
 79  * <ul>
 80  *     <li>closing a native memory segment results in <em>freeing</em> the native memory associated with it</li>
 81  *     <li>closing a mapped memory segment results in the backing memory-mapped file to be unmapped</li>
 82  *     <li>closing a buffer, or a heap segment does not have any side-effect, other than marking the segment
 83  *     as <em>not alive</em> (see {@link MemorySegment#isAlive()}). Also, since the buffer and heap segments might keep
 84  *     strong references to the original buffer or array instance, it is the responsibility of clients to ensure that
 85  *     these segments are discarded in a timely manner, so as not to prevent garbage collection to reclaim the underlying
 86  *     objects.</li>
 87  * </ul>
 88  *
 89  * <h2><a id = "thread-confinement">Thread confinement</a></h2>
 90  *
 91  * Memory segments support strong thread-confinement guarantees. Upon creation, they are assigned an <em>owner thread</em>,
 92  * typically the thread which initiated the creation operation. After creation, only the owner thread will be allowed
 93  * to directly manipulate the memory segment (e.g. close the memory segment) or access the underlying memory associated with
 94  * the segment using a memory access var handle. Any attempt to perform such operations from a thread other than the
 95  * owner thread will result in a runtime failure.
 96  * <p>
 97  * In some cases, it might be useful for multiple threads to process the contents of the same memory segment concurrently
 98  * (e.g. in the case of parallel processing); while memory segments provide strong confinement guarantees, it is possible
 99  * to obtain a {@link Spliterator} from a segment, which can be used to slice the segment and allow multiple thread to
100  * work in parallel on disjoint segment slices (this assumes that the access mode {@link #ACQUIRE} is set).
101  * For instance, the following code can be used to sum all int values in a memory segment in parallel:
102  * <blockquote><pre>{@code
103 SequenceLayout SEQUENCE_LAYOUT = MemoryLayout.ofSequence(1024, MemoryLayouts.JAVA_INT);
104 VarHandle VH_int = SEQUENCE_LAYOUT.elementLayout().varHandle(int.class);
105 int sum = StreamSupport.stream(segment.spliterator(SEQUENCE_LAYOUT), true)
106             .mapToInt(segment -> (int)VH_int.get(segment.baseAddress))
107             .sum();
108  * }</pre></blockquote>
109  *
110  * <h2><a id = "access-modes">Access modes</a></h2>
111  *
112  * Memory segments supports zero or more <em>access modes</em>. Supported access modes are {@link #READ},
113  * {@link #WRITE}, {@link #CLOSE} and {@link #ACQUIRE}. The set of access modes supported by a segment alters the
114  * set of operations that are supported by that segment. For instance, attempting to call {@link #close()} on
115  * a segment which does not support the {@link #CLOSE} access mode will result in an exception.
116  * <p>
117  * The set of supported access modes can only be made stricter (by supporting <em>less</em> access modes). This means
118  * that restricting the set of access modes supported by a segment before sharing it with other clients
119  * is generally a good practice if the creator of the segment wants to retain some control over how the segment
120  * is going to be accessed.
121  *
122  * <h2>Memory segment views</h2>
123  *
124  * Memory segments support <em>views</em>. For instance, it is possible to alter the set of supported access modes,
125  * by creating an <em>immutable</em> view of a memory segment, as follows:
126  * <blockquote><pre>{@code
127 MemorySegment segment = ...
128 MemorySegment roSegment = segment.withAccessModes(segment.accessModes() & ~WRITE);
129  * }</pre></blockquote>
130  * It is also possible to create views whose spatial bounds are stricter than the ones of the original segment
131  * (see {@link MemorySegment#asSlice(long, long)}).
132  * <p>
133  * Temporal bounds of the original segment are inherited by the view; that is, closing a segment view, such as a sliced
134  * view, will cause the original segment to be closed; as such special care must be taken when sharing views
135  * between multiple clients. If a client want to protect itself against early closure of a segment by
136  * another actor, it is the responsibility of that client to take protective measures, such as removing {@link #CLOSE}
137  * from the set of supported access modes, before sharing the view with another client.
138  * <p>
139  * To allow for interoperability with existing code, a byte buffer view can be obtained from a memory segment
140  * (see {@link #asByteBuffer()}). This can be useful, for instance, for those clients that want to keep using the
141  * {@link ByteBuffer} API, but need to operate on large memory segments. Byte buffers obtained in such a way support
142  * the same spatial and temporal access restrictions associated to the memory address from which they originated.
143  *
144  * @apiNote In the future, if the Java language permits, {@link MemorySegment}
145  * may become a {@code sealed} interface, which would prohibit subclassing except by
146  * explicitly permitted types.
147  *
148  * @implSpec
149  * Implementations of this interface are immutable and thread-safe.
150  */
151 public interface MemorySegment extends AutoCloseable {
152 
153     /**
154      * The base memory address associated with this memory segment. The returned address is
155      * a <em>checked</em> memory address and can therefore be used in derefrence operations
156      * (see {@link MemoryAddress}).
157      * @return The base memory address.
158      */
159     MemoryAddress baseAddress();
160 
161     /**
162      * Returns a spliterator for this memory segment. The returned spliterator reports {@link Spliterator#SIZED},
163      * {@link Spliterator#SUBSIZED}, {@link Spliterator#IMMUTABLE}, {@link Spliterator#NONNULL} and {@link Spliterator#ORDERED}
164      * characteristics.
165      * <p>
166      * The returned spliterator splits the segment according to the specified sequence layout; that is,
167      * if the supplied layout is a sequence layout whose element count is {@code N}, then calling {@link Spliterator#trySplit()}
168      * will result in a spliterator serving approximatively {@code N/2} elements (depending on whether N is even or not).
169      * As such, splitting is possible as long as {@code N >= 2}.
170      * <p>
171      * The returned spliterator effectively allows to slice a segment into disjoint sub-segments, which can then
172      * be processed in parallel by multiple threads (if the access mode {@link #ACQUIRE} is set).
173      * While closing the segment (see {@link #close()}) during pending concurrent execution will generally
174      * fail with an exception, it is possible to close a segment when a spliterator has been obtained but no thread
175      * is actively working on it using {@link Spliterator#tryAdvance(Consumer)}; in such cases, any subsequent call
176      * to {@link Spliterator#tryAdvance(Consumer)} will fail with an exception.
177      * @param layout the layout to be used for splitting.
178      * @return the element spliterator for this segment
179      * @throws IllegalStateException if this segment is not <em>alive</em>, or if access occurs from a thread other than the
180      * thread owning this segment
181      */
182     Spliterator<MemorySegment> spliterator(SequenceLayout layout);
183 
184     /**
185      * The thread owning this segment.
186      * @return the thread owning this segment.
187      */
188     Thread ownerThread();
189 
190     /**
191      * The size (in bytes) of this memory segment.
192      * @return The size (in bytes) of this memory segment.
193      */
194     long byteSize();
195 
196     /**
197      * Obtains a segment view with specific <a href="#access-modes">access modes</a>. Supported access modes are {@link #READ}, {@link #WRITE},
198      * {@link #CLOSE} and {@link #ACQUIRE}. It is generally not possible to go from a segment with stricter access modes
199      * to one with less strict access modes. For instance, attempting to add {@link #WRITE} access mode to a read-only segment
200      * will be met with an exception.
201      * @param accessModes an ORed mask of zero or more access modes.
202      * @return a segment view with specific access modes.
203      * @throws UnsupportedOperationException when {@code mask} is an access mask which is less strict than the one supported by this
204      * segment.
205      */
206     MemorySegment withAccessModes(int accessModes);
207 
208     /**
209      * Does this segment support a given set of access modes?
210      * @param accessModes an ORed mask of zero or more access modes.
211      * @return true, if the access modes in {@code accessModes} are stricter than the ones supported by this segment.
212      */
213     boolean hasAccessModes(int accessModes);
214 
215     /**
216      * Returns the <a href="#access-modes">access modes</a> associated with this segment; the result is represented as ORed values from
217      * {@link #READ}, {@link #WRITE}, {@link #CLOSE} and {@link #ACQUIRE}.
218      * @return the access modes associated with this segment.
219      */
220     int accessModes();
221 
222     /**
223      * Obtains a new memory segment view whose base address is the same as the base address of this segment plus a given offset,
224      * and whose new size is specified by the given argument.
225      * @param offset The new segment base offset (relative to the current segment base address), specified in bytes.
226      * @param newSize The new segment size, specified in bytes.
227      * @return a new memory segment view with updated base/limit addresses.
228      * @throws IndexOutOfBoundsException if {@code offset < 0}, {@code offset > byteSize()}, {@code newSize < 0}, or {@code newSize > byteSize() - offset}
229      */
230     MemorySegment asSlice(long offset, long newSize);
231 
232     /**
233      * Is this segment alive?
234      * @return true, if the segment is alive.
235      * @see MemorySegment#close()
236      */
237     boolean isAlive();
238 
239     /**
240      * Closes this memory segment. Once a memory segment has been closed, any attempt to use the memory segment,
241      * or to access the memory associated with the segment will fail with {@link IllegalStateException}. Depending on
242      * the kind of memory segment being closed, calling this method further trigger deallocation of all the resources
243      * associated with the memory segment.
244      * @throws IllegalStateException if this segment is not <em>alive</em>, or if access occurs from a thread other than the
245      * thread owning this segment, or if the segment cannot be closed because it is being operated upon by a different
246      * thread (see {@link #spliterator(SequenceLayout)}).
247      * @throws UnsupportedOperationException if this segment does not support the {@link #CLOSE} access mode.
248      */
249     void close();
250 
251     /**
252      * Wraps this segment in a {@link ByteBuffer}. Some of the properties of the returned buffer are linked to
253      * the properties of this segment. For instance, if this segment is <em>immutable</em>
254      * (e.g. the segment has access mode {@link #READ} but not {@link #WRITE}), then the resulting buffer is <em>read-only</em>
255      * (see {@link ByteBuffer#isReadOnly()}. Additionally, if this is a native memory segment, the resulting buffer is
256      * <em>direct</em> (see {@link ByteBuffer#isDirect()}).
257      * <p>
258      * The life-cycle of the returned buffer will be tied to that of this segment. That means that if the this segment
259      * is closed (see {@link MemorySegment#close()}, accessing the returned
260      * buffer will throw an {@link IllegalStateException}.
261      * <p>
262      * The resulting buffer's byte order is {@link java.nio.ByteOrder#BIG_ENDIAN}; this can be changed using
263      * {@link ByteBuffer#order(java.nio.ByteOrder)}.
264      *
265      * @return a {@link ByteBuffer} view of this memory segment.
266      * @throws UnsupportedOperationException if this segment cannot be mapped onto a {@link ByteBuffer} instance,
267      * e.g. because it models an heap-based segment that is not based on a {@code byte[]}), or if its size is greater
268      * than {@link Integer#MAX_VALUE}, or if the segment does not support the {@link #READ} access mode.
269      */
270     ByteBuffer asByteBuffer();
271 
272     /**
273      * Copy the contents of this memory segment into a fresh byte array.
274      * @return a fresh byte array copy of this memory segment.
275      * @throws UnsupportedOperationException if this segment's contents cannot be copied into a {@link byte[]} instance,
276      * e.g. its size is greater than {@link Integer#MAX_VALUE}.
277      * @throws IllegalStateException if this segment has been closed, or if access occurs from a thread other than the
278      * thread owning this segment.
279      */
280     byte[] toByteArray();

282     /**
283      * Creates a new buffer memory segment that models the memory associated with the given byte
284      * buffer. The segment starts relative to the buffer's position (inclusive)
285      * and ends relative to the buffer's limit (exclusive).
286      * <p>
287      * The resulting memory segment keeps a reference to the backing buffer, to ensure it remains <em>reachable</em>
288      * for the life-time of the segment.
289      *
290      * @param bb the byte buffer backing the buffer memory segment.
291      * @return a new buffer memory segment.
292      */
293     static MemorySegment ofByteBuffer(ByteBuffer bb) {
294         return Utils.makeBufferSegment(bb);
295     }

388     /**
389      * Creates a new native memory segment that models a newly allocated block of off-heap memory with given layout.
390      * <p>
391      * This is equivalent to the following code:
392      * <blockquote><pre>{@code
393     allocateNative(layout.bytesSize(), layout.bytesAlignment());
394      * }</pre></blockquote>
395      *
396      * @implNote The block of off-heap memory associated with the returned native memory segment is initialized to zero.
397      * Moreover, a client is responsible to call the {@link MemorySegment#close()} on a native memory segment,
398      * to make sure the backing off-heap memory block is deallocated accordingly. Failure to do so will result in off-heap memory leaks.
399      *
400      * @param layout the layout of the off-heap memory block backing the native memory segment.
401      * @return a new native memory segment.
402      * @throws IllegalArgumentException if the specified layout has illegal size or alignment constraint.
403      */
404     static MemorySegment allocateNative(MemoryLayout layout) {
405         return allocateNative(layout.byteSize(), layout.byteAlignment());
406     }

408     /**
409      * Creates a new native memory segment that models a newly allocated block of off-heap memory with given size (in bytes).

411      * This is equivalent to the following code:
412      * <blockquote><pre>{@code
413 allocateNative(bytesSize, 1);
414      * }</pre></blockquote>
415      *
416      * @implNote The block of off-heap memory associated with the returned native memory segment is initialized to zero.
417      * Moreover, a client is responsible to call the {@link MemorySegment#close()} on a native memory segment,
418      * to make sure the backing off-heap memory block is deallocated accordingly. Failure to do so will result in off-heap memory leaks.
419      *
420      * @param bytesSize the size (in bytes) of the off-heap memory block backing the native memory segment.
421      * @return a new native memory segment.
422      * @throws IllegalArgumentException if {@code bytesSize < 0}.
423      */
424     static MemorySegment allocateNative(long bytesSize) {
425         return allocateNative(bytesSize, 1);
426     }
427 
428     /**
429      * Creates a new mapped memory segment that models a memory-mapped region of a file from a given path.
430      *
431      * @implNote When obtaining a mapped segment from a newly created file, the initialization state of the contents of the block
432      * of mapped memory associated with the returned mapped memory segment is unspecified and should not be relied upon.
433      *
434      * @param path the path to the file to memory map.
435      * @param bytesSize the size (in bytes) of the mapped memory backing the memory segment.
436      * @param mapMode a file mapping mode, see {@link FileChannel#map(FileChannel.MapMode, long, long)}.
437      * @return a new mapped memory segment.
438      * @throws IllegalArgumentException if {@code bytesSize < 0}.
439      * @throws UnsupportedOperationException if an unsupported map mode is specified.
440      * @throws IOException if the specified path does not point to an existing file, or if some other I/O error occurs.
441      */
442     static MemorySegment mapFromPath(Path path, long bytesSize, FileChannel.MapMode mapMode) throws IOException {
443         return Utils.makeMappedSegment(path, bytesSize, mapMode);
444     }
445 
446     /**
447      * Creates a new native memory segment that models a newly allocated block of off-heap memory with given size and
448      * alignment constraint (in bytes).
449      *
450      * @implNote The block of off-heap memory associated with the returned native memory segment is initialized to zero.
451      * Moreover, a client is responsible to call the {@link MemorySegment#close()} on a native memory segment,
452      * to make sure the backing off-heap memory block is deallocated accordingly. Failure to do so will result in off-heap memory leaks.
453      *
454      * @param bytesSize the size (in bytes) of the off-heap memory block backing the native memory segment.
455      * @param alignmentBytes the alignment constraint (in bytes) of the off-heap memory block backing the native memory segment.
456      * @return a new native memory segment.
457      * @throws IllegalArgumentException if {@code bytesSize < 0}, {@code alignmentBytes < 0}, or if {@code alignmentBytes}
458      * is not a power of 2.
459      */
460     static MemorySegment allocateNative(long bytesSize, long alignmentBytes) {
461         if (bytesSize <= 0) {
462             throw new IllegalArgumentException("Invalid allocation size : " + bytesSize);
463         }
464 
465         if (alignmentBytes < 0 ||
466                 ((alignmentBytes & (alignmentBytes - 1)) != 0L)) {
467             throw new IllegalArgumentException("Invalid alignment constraint : " + alignmentBytes);
468         }
469 
470         return Utils.makeNativeSegment(bytesSize, alignmentBytes);
471     }

502     int ACQUIRE = CLOSE << 1;
