 44 /**
 45  * A memory segment models a contiguous region of memory. A memory segment is associated with both spatial
 46  * and temporal bounds. Spatial bounds ensure that memory access operations on a memory segment cannot affect a memory location
 47  * which falls <em>outside</em> the boundaries of the memory segment being accessed. Temporal checks ensure that memory access
 48  * operations on a segment cannot occur after a memory segment has been closed (see {@link MemorySegment#close()}).
 49  * <p>
 50  * All implementations of this interface must be <a href="{@docRoot}/java.base/java/lang/doc-files/ValueBased.html">value-based</a>;
 51  * use of identity-sensitive operations (including reference equality ({@code ==}), identity hash code, or synchronization) on
 52  * instances of {@code MemorySegment} may have unpredictable results and should be avoided. The {@code equals} method should
 53  * be used for comparisons.
 54  * <p>
 55  * Non-platform classes should not implement {@linkplain MemorySegment} directly.
 56  *
 57  * <h2>Constructing memory segments from different sources</h2>
 58  *
 59  * There are multiple ways to obtain a memory segment. First, memory segments backed by off-heap memory can
 60  * be allocated using one of the many factory methods provided (see {@link MemorySegment#allocateNative(MemoryLayout)},
 61  * {@link MemorySegment#allocateNative(long)} and {@link MemorySegment#allocateNative(long, long)}). Memory segments obtained
 62  * in this way are called <em>native memory segments</em>.
 63  * <p>
 64  * It is also possible to obtain a memory segment backed by an existing heap-allocated Java array,
 65  * using one of the provided factory methods (e.g. {@link MemorySegment#ofArray(int[])}). Memory segments obtained
 66  * in this way are called <em>array memory segments</em>.
 67  * <p>
 68  * It is possible to obtain a memory segment backed by an existing Java byte buffer (see {@link ByteBuffer}),
 69  * using the factory method {@link MemorySegment#ofByteBuffer(ByteBuffer)}.
 70  * Memory segments obtained in this way are called <em>buffer memory segments</em>. Note that buffer memory segments might
 71  * be backed by native memory (as in the case of native memory segments) or heap memory (as in the case of array memory segments),
 72  * depending on the characteristics of the byte buffer instance the segment is associated with. For instance, a buffer memory
 73  * segment obtained from a byte buffer created with the {@link ByteBuffer#allocateDirect(int)} method will be backed
 74  * by native memory.
 75  * <p>
 76  * Finally, it is also possible to obtain a memory segment backed by a memory-mapped file using the factory method
 77  * {@link MemorySegment#mapFromPath(Path, long, long, FileChannel.MapMode)}. Such memory segments are called <em>mapped memory segments</em>
 78  * (see {@link MappedMemorySegment}).
 79  * <p>
 80  * Array and buffer segments are effectively <em>views</em> over existing memory regions which might outlive the
 81  * lifecycle of the segments derived from them, and can even be manipulated directly (e.g. via array access, or direct use
 82  * of the {@link ByteBuffer} API) by other clients. As a result, while sharing array or buffer segments is possible,
 83  * it is strongly advised that clients wishing to do so take extra precautions to make sure that the underlying memory sources
 84  * associated with such segments remain inaccessible, and that said memory sources are never aliased by more than one segment
 85  * at a time - e.g. so as to prevent concurrent modifications of the contents of an array, or buffer segment.
 86  *
 87  * <h2>Closing a memory segment</h2>
 88  *
 89  * Memory segments are closed explicitly (see {@link MemorySegment#close()}). When a segment is closed, it is no longer
 90  * <em>alive</em> (see {@link #isAlive()}, and subsequent operation on the segment (or on any {@link MemoryAddress} instance
 91  * derived from it) will fail with {@link IllegalStateException}.
 92  * <p>
 93  * Closing a segment might trigger the releasing of the underlying memory resources associated with said segment, depending on
 94  * the kind of memory segment being considered:
 95  * <ul>
 96  *     <li>closing a native memory segment results in <em>freeing</em> the native memory associated with it</li>
 97  *     <li>closing a mapped memory segment results in the backing memory-mapped file to be unmapped</li>
 98  *     <li>closing a buffer, or a heap segment does not have any side-effect, other than marking the segment
 99  *     as <em>not alive</em> (see {@link MemorySegment#isAlive()}). Also, since the buffer and heap segments might keep
100  *     strong references to the original buffer or array instance, it is the responsibility of clients to ensure that
101  *     these segments are discarded in a timely manner, so as not to prevent garbage collection to reclaim the underlying
102  *     objects.</li>
103  * </ul>
104  *
105  * <h2><a id = "thread-confinement">Thread confinement</a></h2>
106  *
107  * Memory segments support strong thread-confinement guarantees. Upon creation, they are assigned an <em>owner thread</em>,
108  * typically the thread which initiated the creation operation. After creation, only the owner thread will be allowed
109  * to directly manipulate the memory segment (e.g. close the memory segment) or access the underlying memory associated with
110  * the segment using a memory access var handle. Any attempt to perform such operations from a thread other than the
111  * owner thread will result in a runtime failure.
112  * <p>
113  * Memory segments support <em>serial thread confinement</em>; that is, ownership of a memory segment can change (see
114  * {@link #withOwnerThread(Thread)}). This allows, for instance, for two threads {@code A} and {@code B} to share
115  * a segment in a controlled, cooperative and race-free fashion.
116  * <p>
117  * In some cases, it might be useful for multiple threads to process the contents of the same memory segment concurrently
118  * (e.g. in the case of parallel processing); while memory segments provide strong confinement guarantees, it is possible
119  * to obtain a {@link Spliterator} from a segment, which can be used to slice the segment and allow multiple thread to
120  * work in parallel on disjoint segment slices (this assumes that the access mode {@link #ACQUIRE} is set).
121  * For instance, the following code can be used to sum all int values in a memory segment in parallel:
122  * <blockquote><pre>{@code
123 MemorySegment segment = ...
124 SequenceLayout SEQUENCE_LAYOUT = MemoryLayout.ofSequence(1024, MemoryLayouts.JAVA_INT);
125 VarHandle VH_int = SEQUENCE_LAYOUT.elementLayout().varHandle(int.class);
126 int sum = StreamSupport.stream(MemorySegment.spliterator(segment, SEQUENCE_LAYOUT), true)
127             .mapToInt(s -> (int)VH_int.get(s.baseAddress()))
128             .sum();
129  * }</pre></blockquote>
130  *
131  * <h2><a id = "access-modes">Access modes</a></h2>
132  *
133  * Memory segments supports zero or more <em>access modes</em>. Supported access modes are {@link #READ},
134  * {@link #WRITE}, {@link #CLOSE}, {@link #ACQUIRE} and {@link #HANDOFF}. The set of access modes supported by a segment alters the
135  * set of operations that are supported by that segment. For instance, attempting to call {@link #close()} on
136  * a segment which does not support the {@link #CLOSE} access mode will result in an exception.
137  * <p>
138  * The set of supported access modes can only be made stricter (by supporting <em>fewer</em> access modes). This means
139  * that restricting the set of access modes supported by a segment before sharing it with other clients
140  * is generally a good practice if the creator of the segment wants to retain some control over how the segment
141  * is going to be accessed.
142  *
143  * <h2>Memory segment views</h2>
144  *
145  * Memory segments support <em>views</em>. For instance, it is possible to alter the set of supported access modes,
146  * by creating an <em>immutable</em> view of a memory segment, as follows:
147  * <blockquote><pre>{@code
148 MemorySegment segment = ...
149 MemorySegment roSegment = segment.withAccessModes(segment.accessModes() & ~WRITE);
150  * }</pre></blockquote>
151  * It is also possible to create views whose spatial bounds are stricter than the ones of the original segment
152  * (see {@link MemorySegment#asSlice(long, long)}).
153  * <p>
154  * Temporal bounds of the original segment are inherited by the view; that is, closing a segment view, such as a sliced
155  * view, will cause the original segment to be closed; as such special care must be taken when sharing views
156  * between multiple clients. If a client want to protect itself against early closure of a segment by
157  * another actor, it is the responsibility of that client to take protective measures, such as removing {@link #CLOSE}
158  * from the set of supported access modes, before sharing the view with another client.
159  * <p>
160  * To allow for interoperability with existing code, a byte buffer view can be obtained from a memory segment
161  * (see {@link #asByteBuffer()}). This can be useful, for instance, for those clients that want to keep using the
162  * {@link ByteBuffer} API, but need to operate on large memory segments. Byte buffers obtained in such a way support
163  * the same spatial and temporal access restrictions associated to the memory segment from which they originated.
164  *
165  * @apiNote In the future, if the Java language permits, {@link MemorySegment}
166  * may become a {@code sealed} interface, which would prohibit subclassing except by
167  * {@link MappedMemorySegment} and other explicitly permitted subtypes.
168  *
169  * @implSpec
170  * Implementations of this interface are immutable and thread-safe.
171  */

174     /**
175      * The base memory address associated with this memory segment. The returned address is

182     /**
183      * Returns a spliterator for the given memory segment. The returned spliterator reports {@link Spliterator#SIZED},
184      * {@link Spliterator#SUBSIZED}, {@link Spliterator#IMMUTABLE}, {@link Spliterator#NONNULL} and {@link Spliterator#ORDERED}
185      * characteristics.
186      * <p>
187      * The returned spliterator splits the segment according to the specified sequence layout; that is,
188      * if the supplied layout is a sequence layout whose element count is {@code N}, then calling {@link Spliterator#trySplit()}
189      * will result in a spliterator serving approximatively {@code N/2} elements (depending on whether N is even or not).
190      * As such, splitting is possible as long as {@code N >= 2}. The spliterator returns segments that feature the same
191      * <a href="#access-modes">access modes</a> as the given segment less the {@link #CLOSE} access mode.
192      * <p>
193      * The returned spliterator effectively allows to slice a segment into disjoint sub-segments, which can then
194      * be processed in parallel by multiple threads (if the access mode {@link #ACQUIRE} is set).
195      * While closing the segment (see {@link #close()}) during pending concurrent execution will generally
196      * fail with an exception, it is possible to close a segment when a spliterator has been obtained but no thread
197      * is actively working on it using {@link Spliterator#tryAdvance(Consumer)}; in such cases, any subsequent call
198      * to {@link Spliterator#tryAdvance(Consumer)} will fail with an exception.
199      * @param segment the segment to be used for splitting.
200      * @param layout the layout to be used for splitting.
201      * @param <S> the memory segment type
202      * @return the element spliterator for this segment
203      * @throws IllegalStateException if the segment is not <em>alive</em>, or if access occurs from a thread other than the
204      * thread owning this segment
205      */
206     static <S extends MemorySegment> Spliterator<S> spliterator(S segment, SequenceLayout layout) {
207         return AbstractMemorySegmentImpl.spliterator(segment, layout);
208     }

223      * the new owner thread.
224      *
225      * @param newOwner the new owner thread.
226      * @return a new memory segment backed by the same underlying memory region as this segment,
227      *      owned by {@code newOwner}.
228      * @throws IllegalStateException if this segment is not <em>alive</em>, or if access occurs from a thread other than the

230      * thread (see {@link #spliterator(MemorySegment, SequenceLayout)}).
231      * @throws NullPointerException if {@code newOwner == null}
232      * @throws IllegalArgumentException if the segment is already a confined segment owner by {@code newOnwer}.
233      * @throws UnsupportedOperationException if this segment does not support the {@link #HANDOFF} access mode.
234      */

299     /**
300      * Fills a value into this memory segment.
301      * <p>
302      * More specifically, the given value is filled into each address of this
303      * segment. Equivalent to (but likely more efficient than) the following code:
304      *
305      * <pre>{@code
306 byteHandle = MemoryLayout.ofSequence(MemoryLayouts.JAVA_BYTE)
307          .varHandle(byte.class, MemoryLayout.PathElement.sequenceElement());
308 for (long l = 0; l < segment.byteSize(); l++) {
309      byteHandle.set(segment.baseAddress(), l, value);
310 }
311      * }</pre>
312      *
313      * without any regard or guarantees on the ordering of particular memory
314      * elements being set.
315      * <p>
316      * Fill can be useful to initialize or reset the memory of a segment.
317      *
318      * @param value the value to fill into this segment
319      * @return this memory segment
320      * @throws IllegalStateException if this segment is not <em>alive</em>, or if access occurs from a thread other than the
321      * thread owning this segment
322      * @throws UnsupportedOperationException if this segment does not support the {@link #WRITE} access mode
323      */
324     MemorySegment fill(byte value);

326     /**
327      * Performs a bulk copy from given source segment to this segment. More specifically, the bytes at
328      * offset {@code 0} through {@code src.byteSize() - 1} in the source segment are copied into this segment
329      * at offset {@code 0} through {@code src.byteSize() - 1}.
330      * If the source segment overlaps with this segment, then the copying is performed as if the bytes at
331      * offset {@code 0} through {@code src.byteSize() - 1} in the source segment were first copied into a
332      * temporary segment with size {@code bytes}, and then the contents of the temporary segment were copied into
333      * this segment at offset {@code 0} through {@code src.byteSize() - 1}.
334      * <p>
335      * The result of a bulk copy is unspecified if, in the uncommon case, the source segment and this segment
336      * do not overlap, but refer to overlapping regions of the same backing storage using different addresses.
337      * For example, this may occur if the same file is {@link MemorySegment#mapFromPath mapped} to two segments.
338      *
339      * @param src the source segment.
340      * @throws IndexOutOfBoundsException if {@code src.byteSize() > this.byteSize()}.
341      * @throws IllegalStateException if either the source segment or this segment have been already closed,
342      * or if access occurs from a thread other than the thread owning either segment.
343      * @throws UnsupportedOperationException if either the source segment or this segment do not feature required access modes;
344      * more specifically, {@code src} should feature at least the {@link MemorySegment#READ} access mode,
345      * while this segment should feature at least the {@link MemorySegment#WRITE} access mode.
346      */
347     void copyFrom(MemorySegment src);

349     /**
350      * Finds and returns the offset, in bytes, of the first mismatch between
351      * this segment and a given other segment. The offset is relative to the
352      * {@link #baseAddress() base address} of each segment and will be in the
353      * range of 0 (inclusive) up to the {@link #byteSize() size} (in bytes) of
354      * the smaller memory segment (exclusive).
355      * <p>
356      * If the two segments share a common prefix then the returned offset is
357      * the length of the common prefix and it follows that there is a mismatch
358      * between the two segments at that offset within the respective segments.
359      * If one segment is a proper prefix of the other then the returned offset is
360      * the smaller of the segment sizes, and it follows that the offset is only
361      * valid for the larger segment. Otherwise, there is no mismatch and {@code
362      * -1} is returned.
363      *
364      * @param other the segment to be tested for a mismatch with this segment
365      * @return the relative offset, in bytes, of the first mismatch between this
366      * and the given other segment, otherwise -1 if no mismatch
367      * @throws IllegalStateException if either this segment of the other segment
368      * have been already closed, or if access occurs from a thread other than the
369      * thread owning either segment
370      * @throws UnsupportedOperationException if either this segment or the other
371      * segment does not feature at least the {@link MemorySegment#READ} access mode
372      */
373     long mismatch(MemorySegment other);

375     /**
376      * Wraps this segment in a {@link ByteBuffer}. Some of the properties of the returned buffer are linked to
377      * the properties of this segment. For instance, if this segment is <em>immutable</em>
378      * (e.g. the segment has access mode {@link #READ} but not {@link #WRITE}), then the resulting buffer is <em>read-only</em>
379      * (see {@link ByteBuffer#isReadOnly()}. Additionally, if this is a native memory segment, the resulting buffer is
380      * <em>direct</em> (see {@link ByteBuffer#isDirect()}).
381      * <p>
382      * The life-cycle of the returned buffer will be tied to that of this segment. That means that if the this segment
383      * is closed (see {@link MemorySegment#close()}, accessing the returned
384      * buffer will throw an {@link IllegalStateException}.
385      * <p>
386      * The resulting buffer's byte order is {@link java.nio.ByteOrder#BIG_ENDIAN}; this can be changed using
387      * {@link ByteBuffer#order(java.nio.ByteOrder)}.
388      *
389      * @return a {@link ByteBuffer} view of this memory segment.
390      * @throws UnsupportedOperationException if this segment cannot be mapped onto a {@link ByteBuffer} instance,
391      * e.g. because it models an heap-based segment that is not based on a {@code byte[]}), or if its size is greater
392      * than {@link Integer#MAX_VALUE}, or if the segment does not support the {@link #READ} access mode.
393      */
394     ByteBuffer asByteBuffer();

399      * @throws UnsupportedOperationException if this segment does not feature the {@link #READ} access mode, or if this
400      * segment's contents cannot be copied into a {@link byte[]} instance, e.g. its size is greater than {@link Integer#MAX_VALUE},
401      * @throws IllegalStateException if this segment has been closed, or if access occurs from a thread other than the
402      * thread owning this segment.
403      */

587     /**
588      * Creates a new native memory segment that models a newly allocated block of off-heap memory with given size and
589      * alignment constraint (in bytes). The segment will feature all <a href="#access-modes">access modes</a>
590      * (see {@link #ALL_ACCESS}).
591      *
592      * @implNote The block of off-heap memory associated with the returned native memory segment is initialized to zero.
593      * Moreover, a client is responsible to call the {@link MemorySegment#close()} on a native memory segment,
594      * to make sure the backing off-heap memory block is deallocated accordingly. Failure to do so will result in off-heap memory leaks.
595      *
596      * @param bytesSize the size (in bytes) of the off-heap memory block backing the native memory segment.
597      * @param alignmentBytes the alignment constraint (in bytes) of the off-heap memory block backing the native memory segment.
598      * @return a new native memory segment.
599      * @throws IllegalArgumentException if {@code bytesSize < 0}, {@code alignmentBytes < 0}, or if {@code alignmentBytes}
600      * is not a power of 2.
601      */
602     static MemorySegment allocateNative(long bytesSize, long alignmentBytes) {
603         if (bytesSize <= 0) {
604             throw new IllegalArgumentException("Invalid allocation size : " + bytesSize);
605         }
606 
607         if (alignmentBytes < 0 ||
608                 ((alignmentBytes & (alignmentBytes - 1)) != 0L)) {
609             throw new IllegalArgumentException("Invalid alignment constraint : " + alignmentBytes);
610         }
611 
612         return NativeMemorySegmentImpl.makeNativeSegment(bytesSize, alignmentBytes);
613     }

615     /**
616      * Returns a new native memory segment with given base address and size; the returned segment has its own temporal
617      * bounds, and can therefore be closed; closing such a segment can optionally result in calling an user-provided cleanup
618      * action. This method can be very useful when interacting with custom native memory sources (e.g. custom allocators,
619      * GPU memory, etc.), where an address to some underlying memory region is typically obtained from native code
620      * (often as a plain {@code long} value). The segment will feature all <a href="#access-modes">access modes</a>
621      * (see {@link #ALL_ACCESS}).
622      * <p>
623      * This method is <em>restricted</em>. Restricted method are unsafe, and, if used incorrectly, their use might crash
624      * the JVM or, worse, silently result in memory corruption. Thus, clients should refrain from depending on
625      * restricted methods, and use safe and supported functionalities, where possible.
626      *
627      * @param addr the desired base address
628      * @param bytesSize the desired size.
629      * @param owner the desired owner thread. If {@code owner == null}, the returned segment is <em>not</em> confined.
630      * @param cleanup a cleanup action to be executed when the {@link MemorySegment#close()} method is called on the
631      *                returned segment. If {@code cleanup == null}, no cleanup action is executed.
632      * @param attachment an object that must be kept alive by the returned segment; this can be useful when
633      *                   the returned segment depends on memory which could be released if a certain object
634      *                   is determined to be unreacheable. In most cases this will be set to {@code null}.
635      * @return a new native memory segment with given base address, size, owner, cleanup action and object attachment.
636      * @throws IllegalArgumentException if {@code bytesSize <= 0}.
637      * @throws UnsupportedOperationException if {@code addr} is associated with an heap segment.
638      * @throws IllegalAccessError if the runtime property {@code foreign.restricted} is not set to either
639      * {@code permit}, {@code warn} or {@code debug} (the default value is set to {@code deny}).
640      * @throws NullPointerException if {@code addr == null}.
641      */
642     static MemorySegment ofNativeRestricted(MemoryAddress addr, long bytesSize, Thread owner, Runnable cleanup, Object attachment) {
643         Objects.requireNonNull(addr);
644         if (bytesSize <= 0) {
645             throw new IllegalArgumentException("Invalid size : " + bytesSize);
646         }
647         Utils.checkRestrictedAccess("MemorySegment.ofNativeRestricted");
648         return NativeMemorySegmentImpl.makeNativeSegmentUnchecked(addr, bytesSize, owner, cleanup, attachment);
649     }
