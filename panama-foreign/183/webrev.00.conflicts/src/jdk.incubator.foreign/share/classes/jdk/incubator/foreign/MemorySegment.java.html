 48 /**
 49  * A memory segment models a contiguous region of memory. A memory segment is associated with both spatial
 50  * and temporal bounds. Spatial bounds ensure that memory access operations on a memory segment cannot affect a memory location
 51  * which falls <em>outside</em> the boundaries of the memory segment being accessed. Temporal checks ensure that memory access
 52  * operations on a segment cannot occur after a memory segment has been closed (see {@link MemorySegment#close()}).
 53  * <p>
 54  * All implementations of this interface must be <a href="{@docRoot}/java.base/java/lang/doc-files/ValueBased.html">value-based</a>;
 55  * use of identity-sensitive operations (including reference equality ({@code ==}), identity hash code, or synchronization) on
 56  * instances of {@code MemorySegment} may have unpredictable results and should be avoided. The {@code equals} method should
 57  * be used for comparisons.
 58  * <p>
 59  * Non-platform classes should not implement {@linkplain MemorySegment} directly.
 60  *
 61  * <h2>Constructing memory segments from different sources</h2>
 62  *
 63  * There are multiple ways to obtain a memory segment. First, memory segments backed by off-heap memory can
 64  * be allocated using one of the many factory methods provided (see {@link MemorySegment#allocateNative(MemoryLayout)},
 65  * {@link MemorySegment#allocateNative(long)} and {@link MemorySegment#allocateNative(long, long)}). Memory segments obtained
 66  * in this way are called <em>native memory segments</em>.
 67  * <p>
 68  * It is also possible to obtain a memory segment backed by an existing heap-allocated Java array,
 69  * using one of the provided factory methods (e.g. {@link MemorySegment#ofArray(int[])}). Memory segments obtained
 70  * in this way are called <em>array memory segments</em>.
 71  * <p>
 72  * It is possible to obtain a memory segment backed by an existing Java byte buffer (see {@link ByteBuffer}),
 73  * using the factory method {@link MemorySegment#ofByteBuffer(ByteBuffer)}.
 74  * Memory segments obtained in this way are called <em>buffer memory segments</em>. Note that buffer memory segments might
 75  * be backed by native memory (as in the case of native memory segments) or heap memory (as in the case of array memory segments),
 76  * depending on the characteristics of the byte buffer instance the segment is associated with. For instance, a buffer memory
 77  * segment obtained from a byte buffer created with the {@link ByteBuffer#allocateDirect(int)} method will be backed
 78  * by native memory.
 79  * <p>
 80  * Finally, it is also possible to obtain a memory segment backed by a memory-mapped file using the factory method
 81  * {@link MemorySegment#mapFromPath(Path, long, FileChannel.MapMode)}. Such memory segments are called <em>mapped memory segments</em>
 82  * (see {@link MappedMemorySegment}).
 83  *
 84  * <h2>Closing a memory segment</h2>
 85  *
 86  * Memory segments are closed explicitly (see {@link MemorySegment#close()}). In general when a segment is closed, all off-heap
 87  * resources associated with it are released; this has different meanings depending on the kind of memory segment being
 88  * considered:
 89  * <ul>
 90  *     <li>closing a native memory segment results in <em>freeing</em> the native memory associated with it</li>
 91  *     <li>closing a mapped memory segment results in the backing memory-mapped file to be unmapped</li>
 92  *     <li>closing a buffer, or a heap segment does not have any side-effect, other than marking the segment
 93  *     as <em>not alive</em> (see {@link MemorySegment#isAlive()}). Also, since the buffer and heap segments might keep
 94  *     strong references to the original buffer or array instance, it is the responsibility of clients to ensure that
 95  *     these segments are discarded in a timely manner, so as not to prevent garbage collection to reclaim the underlying
 96  *     objects.</li>
 97  * </ul>
 98  *
 99  * <h2><a id = "thread-confinement">Thread confinement</a></h2>
100  *
101  * Memory segments support strong thread-confinement guarantees. Upon creation, they are assigned an <em>owner thread</em>,
102  * typically the thread which initiated the creation operation. After creation, only the owner thread will be allowed
103  * to directly manipulate the memory segment (e.g. close the memory segment) or access the underlying memory associated with
104  * the segment using a memory access var handle. Any attempt to perform such operations from a thread other than the
105  * owner thread will result in a runtime failure.
106  * <p>
107  * Memory segments support <em>serial thread confinement</em>; that is, ownership of a memory segment can change (see
108  * {@link #withOwnerThread(Thread)}). This allows, for instance, for two threads {@code A} and {@code B} to share
109  * a segment in a controlled, cooperative and race-free fashion.

111  * In some cases, it might be useful for multiple threads to process the contents of the same memory segment concurrently
112  * (e.g. in the case of parallel processing); while memory segments provide strong confinement guarantees, it is possible

152  * <h2>Memory segment views</h2>
153  *
154  * Memory segments support <em>views</em>. For instance, it is possible to alter the set of supported access modes,
155  * by creating an <em>immutable</em> view of a memory segment, as follows:
156  * <blockquote><pre>{@code
157 MemorySegment segment = ...

159  * }</pre></blockquote>
160  * It is also possible to create views whose spatial bounds are stricter than the ones of the original segment
161  * (see {@link MemorySegment#asSlice(long, long)}).
162  * <p>
163  * Temporal bounds of the original segment are inherited by the view; that is, closing a segment view, such as a sliced
164  * view, will cause the original segment to be closed; as such special care must be taken when sharing views
165  * between multiple clients. If a client want to protect itself against early closure of a segment by
166  * another actor, it is the responsibility of that client to take protective measures, such as removing {@link #CLOSE}
167  * from the set of supported access modes, before sharing the view with another client.
168  * <p>
169  * To allow for interoperability with existing code, a byte buffer view can be obtained from a memory segment
170  * (see {@link #asByteBuffer()}). This can be useful, for instance, for those clients that want to keep using the
171  * {@link ByteBuffer} API, but need to operate on large memory segments. Byte buffers obtained in such a way support
172  * the same spatial and temporal access restrictions associated to the memory segment from which they originated.
173  *
174  * @apiNote In the future, if the Java language permits, {@link MemorySegment}
175  * may become a {@code sealed} interface, which would prohibit subclassing except by
176  * {@link MappedMemorySegment} and other explicitly permitted subtypes.
