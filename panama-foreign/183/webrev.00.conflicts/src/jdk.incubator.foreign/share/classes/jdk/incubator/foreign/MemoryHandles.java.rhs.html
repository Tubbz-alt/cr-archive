 44 /**
 45  * This class defines several factory methods for constructing and combining memory access var handles.
 46  * To obtain a memory access var handle, clients must start from one of the <em>leaf</em> methods
 47  * (see {@link MemoryHandles#varHandle(Class, ByteOrder)},
 48  * {@link MemoryHandles#varHandle(Class, long, ByteOrder)}). This determines the variable type
 49  * (all primitive types but {@code void} and {@code boolean} are supported), as well as the alignment constraint and the
 50  * byte order associated to a memory access var handle. The resulting memory access var handle can then be combined in various ways
 51  * to emulate different addressing modes. The var handles created by this class feature a <em>mandatory</em> coordinate type
 52  * (of type {@link MemoryAddress}), and zero or more {@code long} coordinate types, which can be used to emulate
 53  * multi-dimensional array indexing.
 54  * <p>
 55  * As an example, consider the memory layout expressed by a {@link SequenceLayout} instance constructed as follows:
 56  * <blockquote><pre>{@code
 57 SequenceLayout seq = MemoryLayout.ofSequence(5,
 58     MemoryLayout.ofStruct(
 59         MemoryLayout.ofPaddingBits(32),
 60         MemoryLayout.ofValueBits(32, ByteOrder.BIG_ENDIAN).withName("value")
 61     ));
 62  * }</pre></blockquote>
 63  * To access the member layout named {@code value}, we can construct a memory access var handle as follows:
 64  * <blockquote><pre>{@code
 65 VarHandle handle = MemoryHandles.varHandle(int.class, ByteOrder.BIG_ENDIAN); //(MemoryAddress) -> int
 66 handle = MemoryHandles.withOffset(handle, 4); //(MemoryAddress) -> int
 67 handle = MemoryHandles.withStride(handle, 8); //(MemoryAddress, long) -> int
 68  * }</pre></blockquote>
 69  *
 70  * <h2>Addressing mode</h2>
 71  *
 72  * The final memory location accessed by a memory access var handle can be computed as follows:
 73  *
 74  * <blockquote><pre>{@code
 75 address = base + offset
 76  * }</pre></blockquote>
 77  *
 78  * where {@code base} denotes the address expressed by the {@link MemoryAddress} access coordinate, and {@code offset}
 79  * can be expressed in the following form:
 80  *
 81  * <blockquote><pre>{@code
 82 offset = c_1 + c_2 + ... + c_m + (x_1 * s_1) + (x_2 * s_2) + ... + (x_n * s_n)
 83  * }</pre></blockquote>
 84  *
 85  * where {@code x_1}, {@code x_2}, ... {@code x_n} are <em>dynamic</em> values provided as optional {@code long}
 86  * access coordinates, whereas {@code c_1}, {@code c_2}, ... {@code c_m} and {@code s_0}, {@code s_1}, ... {@code s_n} are
 87  * <em>static</em> constants which are can be acquired through the {@link MemoryHandles#withOffset(VarHandle, long)}
 88  * and the {@link MemoryHandles#withStride(VarHandle, long)} combinators, respectively.
 89  *
 90  * <h2><a id="memaccess-mode"></a>Alignment and access modes</h2>
 91  *
 92  * A memory access var handle is associated with an access size {@code S} and an alignment constraint {@code B}
 93  * (both expressed in bytes). We say that a memory access operation is <em>fully aligned</em> if it occurs
 94  * at a memory address {@code A} which is compatible with both alignment constraints {@code S} and {@code B}.
 95  * If access is fully aligned then following access modes are supported and are
 96  * guaranteed to support atomic access:
 97  * <ul>
 98  * <li>read write access modes for all {@code T}, with the exception of
 99  *     access modes {@code get} and {@code set} for {@code long} and
100  *     {@code double} on 32-bit platforms.
101  * <li>atomic update access modes for {@code int}, {@code long},
102  *     {@code float} or {@code double}.
103  *     (Future major platform releases of the JDK may support additional
104  *     types for certain currently unsupported access modes.)
105  * <li>numeric atomic update access modes for {@code int} and {@code long}.
106  *     (Future major platform releases of the JDK may support additional
107  *     numeric types for certain currently unsupported access modes.)
108  * <li>bitwise atomic update access modes for {@code int} and {@code long}.
109  *     (Future major platform releases of the JDK may support additional
110  *     numeric types for certain currently unsupported access modes.)
111  * </ul>
112  *
113  * If {@code T} is {@code float} or {@code double} then atomic
114  * update access modes compare values using their bitwise representation
115  * (see {@link Float#floatToRawIntBits} and
116  * {@link Double#doubleToRawLongBits}, respectively).
117  * <p>
118  * Alternatively, a memory access operation is <em>partially aligned</em> if it occurs at a memory address {@code A}
119  * which is only compatible with the alignment constraint {@code B}; in such cases, access for anything other than the
120  * {@code get} and {@code set} access modes will result in an {@code IllegalStateException}. If access is partially aligned,
121  * atomic access is only guaranteed with respect to the largest power of two that divides the GCD of {@code A} and {@code S}.
122  * <p>
123  * Finally, in all other cases, we say that a memory access operation is <em>misaligned</em>; in such cases an
124  * {@code IllegalStateException} is thrown, irrespective of the access mode being used.
125  */

408     /**
409      * Adapts a target var handle by narrowing incoming values and widening
410      * outgoing values, to and from the given type, respectively.
411      * <p>
412      * The returned var handle can be used to conveniently treat unsigned
413      * primitive data types as if they were a wider signed primitive type. For
414      * example, it is often convenient to model an <i>unsigned short</i> as a
415      * Java {@code int} to avoid dealing with negative values, which would be
416      * the case if modeled as a Java {@code short}. The returned var handle
417      * converts to and from wider primitive types, to a more narrow possibly
418      * unsigned primitive type.
419      * <p>
420      * When calling e.g. {@link VarHandle#set(Object...)} on the resulting var
421      * handle, the incoming value (of type {@code adaptedType}) is converted by a
422      * <i>narrowing primitive conversion</i> and then passed to the {@code
423      * target} var handle. A narrowing primitive conversion may lose information
424      * about the overall magnitude of a numeric value. Conversely, when calling
425      * e.g. {@link VarHandle#get(Object...)} on the resulting var handle, the
426      * returned value obtained from the {@code target} var handle is converted
427      * by a <i>unsigned widening conversion</i> before being returned to the
428      * caller. In an unsigned widening conversion the high-order bits greater
429      * than that of the {@code target} carrier type are zero, and the low-order
430      * bits (equal to the width of the {@code target} carrier type) are equal to
431      * the bits of the value obtained from the {@code target} var handle.
432      * <p>
433      * The returned var handle will feature the variable type {@code adaptedType},
434      * and the same access coordinates, the same access modes (see {@link
435      * java.lang.invoke.VarHandle.AccessMode}, and the same atomic access
436      * guarantees, as those featured by the {@code target} var handle.
437      *
438      * @param target the memory access var handle to be adapted
439      * @param adaptedType the adapted type
440      * @returns the adapted var handle.
441      * @throws IllegalArgumentException if the carrier type of {@code target}
442      * is not one of {@code byte}, {@code short}, or {@code int}; if {@code
443      * adaptedType} is not one of {@code int}, or {@code long}; if the bitwidth
444      * of the {@code adaptedType} is not greater than that of the {@code target}
445      * carrier type
446      * @throws NullPointerException if either of {@code target} or {@code
447      * adaptedType} is null

476      * <p>
477      * When calling e.g. {@link VarHandle#set(Object...)} on the resulting var handle, the incoming value (of type {@code T}, where
478      * {@code T} is the <em>last</em> parameter type of the first filter function) is processed using the first filter and then passed

480      * Conversely, when calling e.g. {@link VarHandle#get(Object...)} on the resulting var handle, the return value obtained from
481      * the target var handle (of type {@code T}, where {@code T} is the <em>last</em> parameter type of the second filter function)
482      * is processed using the second filter and returned to the caller. More advanced access mode types, such as
483      * {@link java.lang.invoke.VarHandle.AccessMode#COMPARE_AND_EXCHANGE} might apply both filters at the same time.
484      * <p>
485      * For the boxing and unboxing filters to be well formed, their types must be of the form {@code (A... , S) -> T} and
486      * {@code (A... , T) -> S}, respectively, where {@code T} is the type of the target var handle. If this is the case,
487      * the resulting var handle will have type {@code S} and will feature the additional coordinates {@code A...} (which
488      * will be appended to the coordinates of the target var handle).
489      * <p>
490      * The resulting var handle will feature the same access modes (see {@link java.lang.invoke.VarHandle.AccessMode} and
491      * atomic access guarantees as those featured by the target var handle.
492      *
493      * @param target the target var handle
494      * @param filterToTarget a filter to convert some type {@code S} into the type of {@code target}
