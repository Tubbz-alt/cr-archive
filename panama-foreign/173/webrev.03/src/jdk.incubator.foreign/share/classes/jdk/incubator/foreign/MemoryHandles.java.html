 42 /**
 43  * This class defines several factory methods for constructing and combining memory access var handles.
 44  * To obtain a memory access var handle, clients must start from one of the <em>leaf</em> methods
 45  * (see {@link MemoryHandles#varHandle(Class, ByteOrder)},
 46  * {@link MemoryHandles#varHandle(Class, long, ByteOrder)}). This determines the variable type
 47  * (all primitive types but {@code void} and {@code boolean} are supported), as well as the alignment constraint and the
 48  * byte order associated to a memory access var handle. The resulting memory access var handle can then be combined in various ways
 49  * to emulate different addressing modes. The var handles created by this class feature a <em>mandatory</em> coordinate type
 50  * (of type {@link MemoryAddress}), and zero or more {@code long} coordinate types, which can be used to emulate
 51  * multi-dimensional array indexing.
 52  * <p>
 53  * As an example, consider the memory layout expressed by a {@link SequenceLayout} instance constructed as follows:
 54  * <blockquote><pre>{@code
 55 SequenceLayout seq = MemoryLayout.ofSequence(5,
 56     MemoryLayout.ofStruct(
 57         MemoryLayout.ofPaddingBits(32),
 58         MemoryLayout.ofValueBits(32, ByteOrder.BIG_ENDIAN).withName("value")
 59     ));
 60  * }</pre></blockquote>
 61  * To access the member layout named {@code value}, we can construct a memory access var handle as follows:
 62  * <blockquote><pre>{@code
 63 VarHandle handle = MemoryHandles.varHandle(int.class, ByteOrder.BIG_ENDIAN); //(MemoryAddress) -> int
 64 handle = MemoryHandles.withOffset(handle, 4); //(MemoryAddress) -> int
 65 handle = MemoryHandles.withStride(handle, 8); //(MemoryAddress, long) -> int
 66  * }</pre></blockquote>
 67  *
 68  * <h2>Addressing mode</h2>
 69  *
 70  * The final memory location accessed by a memory access var handle can be computed as follows:
 71  *
 72  * <blockquote><pre>{@code
 73 address = base + offset
 74  * }</pre></blockquote>
 75  *
 76  * where {@code base} denotes the address expressed by the {@link MemoryAddress} access coordinate, and {@code offset}
 77  * can be expressed in the following form:
 78  *
 79  * <blockquote><pre>{@code
 80 offset = c_1 + c_2 + ... + c_m + (x_1 * s_1) + (x_2 * s_2) + ... + (x_n * s_n)
 81  * }</pre></blockquote>
 82  *
 83  * where {@code x_1}, {@code x_2}, ... {@code x_n} are <em>dynamic</em> values provided as optional {@code long}
 84  * access coordinates, whereas {@code c_1}, {@code c_2}, ... {@code c_m} and {@code s_0}, {@code s_1}, ... {@code s_n} are
 85  * <em>static</em> constants which are can be acquired through the {@link MemoryHandles#withOffset(VarHandle, long)}
 86  * and the {@link MemoryHandles#withStride(VarHandle, long)} combinators, respectively.
 87  *
 88  * <h2><a id="memaccess-mode"></a>Alignment and access modes</h2>
 89  *
 90  * A memory access var handle is associated with an access size {@code S} and an alignment constraint {@code B}
 91  * (both expressed in bytes). We say that a memory access operation is <em>fully aligned</em> if it occurs
 92  * at a memory address {@code A} which is compatible with both alignment constraints {@code S} and {@code B}.
 93  * If access is fully aligned then following access modes are supported and are
 94  * guaranteed to support atomic access:
 95  * <ul>
 96  * <li>read write access modes for all {@code T}, with the exception of
 97  *     access modes {@code get} and {@code set} for {@code long} and
 98  *     {@code double} on 32-bit platforms.
 99  * <li>atomic update access modes for {@code int}, {@code long},
100  *     {@code float} or {@code double}.
101  *     (Future major platform releases of the JDK may support additional
102  *     types for certain currently unsupported access modes.)
103  * <li>numeric atomic update access modes for {@code int} and {@code long}.
104  *     (Future major platform releases of the JDK may support additional
105  *     numeric types for certain currently unsupported access modes.)
106  * <li>bitwise atomic update access modes for {@code int} and {@code long}.
107  *     (Future major platform releases of the JDK may support additional
108  *     numeric types for certain currently unsupported access modes.)
109  * </ul>
110  *
111  * If {@code T} is {@code float} or {@code double} then atomic
112  * update access modes compare values using their bitwise representation
113  * (see {@link Float#floatToRawIntBits} and
114  * {@link Double#doubleToRawLongBits}, respectively).
115  * <p>
116  * Alternatively, a memory access operation is <em>partially aligned</em> if it occurs at a memory address {@code A}
117  * which is only compatible with the alignment constraint {@code B}; in such cases, access for anything other than the
118  * {@code get} and {@code set} access modes will result in an {@code IllegalStateException}. If access is partially aligned,
119  * atomic access is only guaranteed with respect to the largest power of two that divides the GCD of {@code A} and {@code S}.
120  * <p>
121  * Finally, in all other cases, we say that a memory access operation is <em>misaligned</em>; in such cases an
122  * {@code IllegalStateException} is thrown, irrespective of the access mode being used.
123  */

185     /**
186      * Creates a memory access var handle with the given carrier type and byte order.
187      *
188      * The resulting memory access var handle features a single {@link MemoryAddress} access coordinate,
189      * and its variable type is set by the given carrier type.
190      *
191      * The alignment constraint for the resulting memory access var handle is the same as the in memory size of the
192      * carrier type, and the accessed offset is set at zero.
193      *
194      * @apiNote the resulting var handle features certain <a href="#memaccess-mode">access mode restrictions</a>,
195      * which are common to all memory access var handles.
196      *
197      * @param carrier the carrier type. Valid carriers are {@code byte}, {@code short}, {@code char}, {@code int},
198      * {@code float}, {@code long}, and {@code double}.
199      * @param byteOrder the required byte order.
200      * @return the new memory access var handle.
201      * @throws IllegalArgumentException when an illegal carrier type is used
202      */
203     public static VarHandle varHandle(Class<?> carrier, ByteOrder byteOrder) {
204         checkCarrier(carrier);
205         return varHandle(carrier,
206                 carrierSize(carrier),
207                 byteOrder);
208     }

210     /**
211      * Creates a memory access var handle with the given carrier type, alignment constraint, and byte order.
212      *
213      * The resulting memory access var handle features a single {@link MemoryAddress} access coordinate,
214      * and its variable type is set by the given carrier type.
215      *
216      * The accessed offset is zero.
217      *
218      * @apiNote the resulting var handle features certain <a href="#memaccess-mode">access mode restrictions</a>,
219      * which are common to all memory access var handles.
220      *
221      * @param carrier the carrier type. Valid carriers are {@code byte}, {@code short}, {@code char}, {@code int},
222      * {@code float}, {@code long}, and {@code double}.
223      * @param alignmentBytes the alignment constraint (in bytes). Must be a power of two.
224      * @param byteOrder the required byte order.
225      * @return the new memory access var handle.
226      * @throws IllegalArgumentException if an illegal carrier type is used, or if {@code alignmentBytes} is not a power of two.
227      */
228     public static VarHandle varHandle(Class<?> carrier, long alignmentBytes, ByteOrder byteOrder) {
229         checkCarrier(carrier);
230 
231         if (alignmentBytes <= 0
232                 || (alignmentBytes & (alignmentBytes - 1)) != 0) { // is power of 2?
233             throw new IllegalArgumentException("Bad alignment: " + alignmentBytes);
234         }
235 
236         return Utils.fixUpVarHandle(JLI.memoryAccessVarHandle(carrier, alignmentBytes - 1, byteOrder, 0, new long[]{}));
237     }

239     /**
240      * Returns a var handle that adds a <em>fixed</em> offset to the incoming {@link MemoryAddress} coordinate
241      * and then propagates such value to the target var handle. That is,
242      * when the returned var handle receives a memory address coordinate pointing at a memory location at
243      * offset <em>O</em>, a memory address coordinate pointing at a memory location at offset <em>O' + O</em>
244      * is created, and then passed to the target var handle.
245      *
246      * The returned var handle will feature the same type and access coordinates as the target var handle.
247      *
248      * @param target the target memory access handle to access after the offset adjustment.
249      * @param bytesOffset the offset, in bytes. Must be positive or zero.
250      * @return the adapted var handle.
251      * @throws IllegalArgumentException if the first access coordinate type is not of type {@link MemoryAddress}.
252      */
253     public static VarHandle withOffset(VarHandle target, long bytesOffset) {
254         if (bytesOffset == 0) {
255             return target; //nothing to do
256         }
257 
258         checkAddressFirstCoordinate(target);
259 
260         if (JLI.isMemoryAccessVarHandle(target) &&
261                 (bytesOffset & JLI.memoryAddressAlignmentMask(target)) == 0) {
262             //flatten
263             return Utils.fixUpVarHandle(JLI.memoryAccessVarHandle(
264                     JLI.memoryAddressCarrier(target),
265                     JLI.memoryAddressAlignmentMask(target),
266                     JLI.memoryAddressByteOrder(target),
267                     JLI.memoryAddressOffset(target) + bytesOffset,
268                     JLI.memoryAddressStrides(target)));
269         } else {
270             //slow path
271             VarHandle res = collectCoordinates(target, 0, ADD_OFFSET);
272             return insertCoordinates(res, 1, bytesOffset);
273         }
274     }

276     /**
277      * Returns a var handle which adds a <em>variable</em> offset to the incoming {@link MemoryAddress}
278      * access coordinate value and then propagates such value to the target var handle.
279      * That is, when the returned var handle receives a memory address coordinate pointing at a memory location at
280      * offset <em>O</em>, a new memory address coordinate pointing at a memory location at offset <em>(S * X) + O</em>
281      * is created, and then passed to the target var handle,
282      * where <em>S</em> is a constant <em>stride</em>, whereas <em>X</em> is a dynamic value that will be
283      * provided as an additional access coordinate (of type {@code long}).
284      *
285      * The returned var handle will feature the same type as the target var handle; an additional access coordinate
286      * of type {@code long} will be added to the access coordinate types of the target var handle at the position
287      * immediately following the leading access coordinate of type {@link MemoryAddress}.
288      *
289      * @param target the target memory access handle to access after the scale adjustment.
290      * @param bytesStride the stride, in bytes, by which to multiply the coordinate value. Must be greater than zero.
291      * @return the adapted var handle.
292      * @throws IllegalArgumentException if the first access coordinate type is not of type {@link MemoryAddress}.
293      */
294     public static VarHandle withStride(VarHandle target, long bytesStride) {
295         if (bytesStride == 0) {
296             return dropCoordinates(target, 1, long.class); // dummy coordinate
297         }
298 
299         checkAddressFirstCoordinate(target);
300 
301         if (JLI.isMemoryAccessVarHandle(target) &&
302                 (bytesStride & JLI.memoryAddressAlignmentMask(target)) == 0) {
303             //flatten
304             long[] strides = JLI.memoryAddressStrides(target);
305             long[] newStrides = new long[strides.length + 1];
306             System.arraycopy(strides, 0, newStrides, 1, strides.length);
307             newStrides[0] = bytesStride;
308 
309             return Utils.fixUpVarHandle(JLI.memoryAccessVarHandle(
310                     JLI.memoryAddressCarrier(target),
311                     JLI.memoryAddressAlignmentMask(target),
312                     JLI.memoryAddressByteOrder(target),
313                     JLI.memoryAddressOffset(target),
314                     newStrides));
315         } else {
316             //slow path
317             VarHandle res = collectCoordinates(target, 0, ADD_STRIDE);
318             return insertCoordinates(res, 2, bytesStride);
319         }
320     }

322     /**
323      * Adapt an existing var handle into a new var handle whose carrier type is {@link MemoryAddress}.
324      * That is, when calling {@link VarHandle#get(Object...)} on the returned var handle,
325      * the read numeric value will be turned into a memory address (as if by calling {@link MemoryAddress#ofLong(long)});
326      * similarly, when calling {@link VarHandle#set(Object...)}, the memory address to be set will be converted
327      * into a numeric value, and then written into memory. The amount of bytes read (resp. written) from (resp. to)
328      * memory depends on the carrier of the original memory access var handle.
329      *
330      * @param target the memory access var handle to be adapted
331      * @return the adapted var handle.
332      * @throws IllegalArgumentException if the carrier type of {@code varHandle} is either {@code boolean},
333      * {@code float}, or {@code double}, or is not a primitive type.
334      */
335     public static VarHandle asAddressVarHandle(VarHandle target) {
336         Class<?> carrier = target.varType();
337         if (!carrier.isPrimitive() || carrier == boolean.class ||
338                 carrier == float.class || carrier == double.class) {
339             throw new IllegalArgumentException("Unsupported carrier type: " + carrier.getName());
340         }
341 
342         if (carrier != long.class) {
343             // slow-path, we need to adapt
344             return filterValue(target,
345                     MethodHandles.explicitCastArguments(ADDRESS_TO_LONG, MethodType.methodType(carrier, MemoryAddress.class)),
346                     MethodHandles.explicitCastArguments(LONG_TO_ADDRESS, MethodType.methodType(MemoryAddress.class, carrier)));
347         } else {
348             // fast-path
349             return filterValue(target, ADDRESS_TO_LONG, LONG_TO_ADDRESS);
350         }
351     }

353     /**
354      * Adapts a target var handle by narrowing incoming values and widening
355      * outgoing values, to and from the given type, respectively.
356      * <p>
357      * The returned var handle can be used to conveniently treat unsigned
358      * primitive data types as if they were a wider signed primitive type. For
359      * example, it is often convenient to model an <i>unsigned short</i> as a
360      * Java {@code int} to avoid dealing with negative values, which would be
361      * the case if modeled as a Java {@code short}. The returned var handle
362      * converts to and from wider primitive types, to a more narrow possibly
363      * unsigned primitive type.
364      * <p>
365      * When calling e.g. {@link VarHandle#set(Object...)} on the resulting var
366      * handle, the incoming value (of type {@code adaptedType}) is converted by a
367      * <i>narrowing primitive conversion</i> and then passed to the {@code
368      * target} var handle. A narrowing primitive conversion may lose information
369      * about the overall magnitude of a numeric value. Conversely, when calling
370      * e.g. {@link VarHandle#get(Object...)} on the resulting var handle, the
371      * returned value obtained from the {@code target} var handle is converted
372      * by a <i>unsigned widening conversion</i> before being returned to the
373      * caller. In an unsigned widening conversion the high-order bits greater
374      * than that of the {@code target} carrier type are zero, and the low-order
375      * bits (equal to the width of the {@code target} carrier type) are equal to
376      * the bits of the value obtained from the {@code target} var handle.
377      * <p>
378      * The returned var handle will feature the variable type {@code adaptedType},
379      * and the same access coordinates, the same access modes (see {@link
380      * java.lang.invoke.VarHandle.AccessMode}, and the same atomic access
381      * guarantees, as those featured by the {@code target} var handle.
382      *
383      * @param target the memory access var handle to be adapted
384      * @param adaptedType the adapted type
385      * @returns the adapted var handle.
386      * @throws IllegalArgumentException if the carrier type of {@code target}
387      * is not one of {@code byte}, {@code short}, or {@code int}; if {@code
388      * adaptedType} is not one of {@code int}, or {@code long}; if the bitwidth
389      * of the {@code adaptedType} is not greater than that of the {@code target}
390      * carrier type
391      * @throws NullPointerException if either of {@code target} or {@code
392      * adaptedType} is null
393      *
394      * @jls 5.1.3 Narrowing Primitive Conversion
395      */
396     public static VarHandle asUnsigned(VarHandle target, final Class<?> adaptedType) {
397         Objects.requireNonNull(target);
398         Objects.requireNonNull(adaptedType);
399         final Class<?> carrier = target.varType();
400         checkWidenable(carrier);
401         checkNarrowable(adaptedType);
402         checkTargetWiderThanCarrier(carrier, adaptedType);
403 
404         if (adaptedType == int.class && carrier == byte.class) {
405             return filterValue(target, INT_TO_BYTE, BYTE_TO_UNSIGNED_INT);
406         } else if (adaptedType == int.class && carrier == short.class) {
407             return filterValue(target, INT_TO_SHORT, SHORT_TO_UNSIGNED_INT);
408         } else if (adaptedType == long.class && carrier == byte.class) {
409             return filterValue(target, LONG_TO_BYTE, BYTE_TO_UNSIGNED_LONG);
410         } else if (adaptedType == long.class && carrier == short.class) {
411             return filterValue(target, LONG_TO_SHORT, SHORT_TO_UNSIGNED_LONG);
412         } else if (adaptedType == long.class && carrier == int.class) {
413             return filterValue(target, LONG_TO_INT, INT_TO_UNSIGNED_LONG);
414         } else {
415             throw new InternalError("should not reach here");
416         }
417     }

419     /**
420      * Adapts a target var handle by pre-processing incoming and outgoing values using a pair of unary filter functions.
421      * <p>
422      * When calling e.g. {@link VarHandle#set(Object...)} on the resulting var handle, the incoming value (of type {@code T}, where
423      * {@code T} is the parameter type of the first filter function) is processed using the first filter and then passed
424      * to the target var handle.
425      * Conversely, when calling e.g. {@link VarHandle#get(Object...)} on the resulting var handle, the return value obtained from
426      * the target var handle (of type {@code T}, where {@code T} is the parameter type of the second filter function)
427      * is processed using the second filter and returned to the caller. More advanced access mode types, such as
428      * {@link java.lang.invoke.VarHandle.AccessMode#COMPARE_AND_EXCHANGE} might apply both filters at the same time.
429      * <p>
430      * For the boxing and unboxing filters to be well formed, their types must be of the form {@code S -> T} and {@code T -> S},
431      * respectively, where {@code T} is the type of the target var handle. If this is the case, the resulting var handle will
432      * have type {@code S}.
433      * <p>
434      * The resulting var handle will feature the same access modes (see {@link java.lang.invoke.VarHandle.AccessMode} and
435      * atomic access guarantees as those featured by the target var handle.
436      *
437      * @param target the target var handle
438      * @param filterToTarget a filter to convert some type {@code S} into the type of {@code target}
439      * @param filterFromTarget a filter to convert the type of {@code target} to some type {@code S}
440      * @return an adapter var handle which accepts a new type, performing the provided boxing/unboxing conversions.
441      * @throws NullPointerException if either {@code target}, {@code filterToTarget} or {@code filterFromTarget} are {@code == null}.
442      * @throws IllegalArgumentException if {@code filterFromTarget} and {@code filterToTarget} are not well-formed, that is, they have types
443      * other than {@code S -> T} and {@code T -> S}, respectively, where {@code T} is the type of the target var handle,
444      * or if either {@code filterFromTarget} or {@code filterToTarget} throws any checked exceptions.
445      */
446     public static VarHandle filterValue(VarHandle target, MethodHandle filterToTarget, MethodHandle filterFromTarget) {
447         return JLI.filterValue(target, filterToTarget, filterFromTarget);
448     }

450     /**
451      * Adapts a target var handle by pre-processing incoming coordinate values using unary filter functions.
452      * <p>
453      * When calling e.g. {@link VarHandle#get(Object...)} on the resulting var handle, the incoming coordinate values
454      * starting at position {@code pos} (of type {@code C1, C2 ... Cn}, where {@code C1, C2 ... Cn} are the return type
455      * of the unary filter functions) are transformed into new values (of type {@code S1, S2 ... Sn}, where {@code S1, S2 ... Sn} are the
456      * parameter types of the unary filter functions), and then passed (along with any coordinate that was left unaltered
457      * by the adaptation) to the target var handle.
458      * <p>
459      * For the coordinate filters to be well formed, their types must be of the form {@code S1 -> T1, S2 -> T1 ... Sn -> Tn},
460      * where {@code T1, T2 ... Tn} are the coordinate types starting at position {@code pos} of the target var handle.
461      * <p>
462      * The resulting var handle will feature the same access modes (see {@link java.lang.invoke.VarHandle.AccessMode} and
463      * atomic access guarantees as those featured by the target var handle.
464      *
465      * @param target the target var handle
466      * @param pos the position of the first coordinate to be transformed
467      * @param filters the unary functions which are used to transform coordinates starting at position {@code pos}
468      * @return an adapter var handle which accepts new coordinate types, applying the provided transformation
469      * to the new coordinate values.
470      * @throws NullPointerException if either {@code target}, {@code filters} are {@code == null}.
471      * @throws IllegalArgumentException if the handles in {@code filters} are not well-formed, that is, they have types
472      * other than {@code S1 -> T1, S2 -> T2, ... Sn -> Tn} where {@code T1, T2 ... Tn} are the coordinate types starting
473      * at position {@code pos} of the target var handle, if {@code pos} is not between 0 and the target var handle coordinate arity, inclusive,
474      * or if more filters are provided than the actual number of coordinate types, or if any of the filters throws any
475      * checked exceptions.
476      * available starting at {@code pos}.
477      */
478     public static VarHandle filterCoordinates(VarHandle target, int pos, MethodHandle... filters) {
479         return JLI.filterCoordinates(target, pos, filters);
480     }

482     /**
483      * Provides a target var handle with one or more <em>bound coordinates</em>
484      * in advance of the var handle's invocation. As a consequence, the resulting var handle will feature less
485      * coordinate types than the target var handle.
486      * <p>
487      * When calling e.g. {@link VarHandle#get(Object...)} on the resulting var handle, incoming coordinate values
488      * are joined with bound coordinate values, and then passed to the target var handle.
489      * <p>
490      * For the bound coordinates to be well formed, their types must be {@code T1, T2 ... Tn },
491      * where {@code T1, T2 ... Tn} are the coordinate types starting at position {@code pos} of the target var handle.
492      * <p>
493      * The resulting var handle will feature the same access modes (see {@link java.lang.invoke.VarHandle.AccessMode} and
494      * atomic access guarantees as those featured by the target var handle.
495      *
496      * @param target the var handle to invoke after the bound coordinates are inserted
497      * @param pos the position of the first coordinate to be inserted
498      * @param values the series of bound coordinates to insert
499      * @return an adapter var handle which inserts an additional coordinates,
500      *         before calling the target var handle
501      * @throws NullPointerException if either {@code target}, {@code values} are {@code == null}.
502      * @throws IllegalArgumentException if {@code pos} is not between 0 and the target var handle coordinate arity, inclusive,
503      * or if more values are provided than the actual number of coordinate types available starting at {@code pos}.
504      * @throws ClassCastException if the bound coordinates in {@code values} are not well-formed, that is, they have types
505      * other than {@code T1, T2 ... Tn }, where {@code T1, T2 ... Tn} are the coordinate types starting at position {@code pos}
506      * of the target var handle.
507      */
508     public static VarHandle insertCoordinates(VarHandle target, int pos, Object... values) {
509         return JLI.insertCoordinates(target, pos, values);
510     }

512     /**
513      * Provides a var handle which adapts the coordinate values of the target var handle, by re-arranging them
514      * so that the new coordinates match the provided ones.
515      * <p>
516      * The given array controls the reordering.
517      * Call {@code #I} the number of incoming coordinates (the value
518      * {@code newCoordinates.size()}, and call {@code #O} the number
519      * of outgoing coordinates (the number of coordinates associated with the target var handle).
520      * Then the length of the reordering array must be {@code #O},
521      * and each element must be a non-negative number less than {@code #I}.
522      * For every {@code N} less than {@code #O}, the {@code N}-th
523      * outgoing coordinate will be taken from the {@code I}-th incoming
524      * coordinate, where {@code I} is {@code reorder[N]}.
525      * <p>
526      * No coordinate value conversions are applied.
527      * The type of each incoming coordinate, as determined by {@code newCoordinates},
528      * must be identical to the type of the corresponding outgoing coordinate
529      * in the target var handle.
530      * <p>
531      * The reordering array need not specify an actual permutation.
532      * An incoming coordinate will be duplicated if its index appears
533      * more than once in the array, and an incoming coordinate will be dropped
534      * if its index does not appear in the array.
535      * <p>
536      * The resulting var handle will feature the same access modes (see {@link java.lang.invoke.VarHandle.AccessMode} and
537      * atomic access guarantees as those featured by the target var handle.
538      * @param target the var handle to invoke after the coordinates have been reordered
539      * @param newCoordinates the new coordinate types
540      * @param reorder an index array which controls the reordering
541      * @return an adapter var handle which re-arranges the incoming coordinate values,
542      * before calling the target var handle
543      * @throws NullPointerException if either {@code target}, {@code newCoordinates} or {@code reorder} are {@code == null}.
544      * @throws IllegalArgumentException if the index array length is not equal to
545      * the number of coordinates of the target var handle, or if any index array element is not a valid index for
546      * a coordinate of {@code newCoordinates}, or if two corresponding coordinate types in
547      * the target var handle and in {@code newCoordinates} are not identical.
548      */
549     public static VarHandle permuteCoordinates(VarHandle target, List<Class<?>> newCoordinates, int... reorder) {
550         return JLI.permuteCoordinates(target, newCoordinates, reorder);
551     }

553     /**
554      * Adapts a target var handle handle by pre-processing
555      * a sub-sequence of its coordinate values with a filter (a method handle).
556      * The pre-processed coordinates are replaced by the result (if any) of the
557      * filter function and the target var handle is then called on the modified (usually shortened)
558      * coordinate list.
559      * <p>
560      * If {code R} is the return type of the filter (which cannot be void), the target var handle must accept a value of
561      * type {@code R} as its coordinate in position {@code pos}, preceded and/or followed by
562      * any coordinate not passed to the filter.
563      * No coordinates are reordered, and the result returned from the filter
564      * replaces (in order) the whole subsequence of coordinates originally
565      * passed to the adapter.
566      * <p>
567      * The argument types (if any) of the filter
568      * replace zero or one coordinate types of the target var handle, at position {@code pos},
569      * in the resulting adapted var handle.
570      * The return type of the filter must be identical to the
571      * coordinate type of the target var handle at position {@code pos}, and that target var handle
572      * coordinate is supplied by the return value of the filter.
573      * <p>
574      * The resulting var handle will feature the same access modes (see {@link java.lang.invoke.VarHandle.AccessMode} and
575      * atomic access guarantees as those featured by the target var handle.
576      *
577      * @param target the var handle to invoke after the coordinates have been filtered
578      * @param pos the position of the coordinate to be filtered
579      * @param filter the filter method handle
580      * @return an adapter var handle which filters the incoming coordinate values,
581      * before calling the target var handle
582      * @throws NullPointerException if either {@code target}, {@code filter} are {@code == null}.
583      * @throws IllegalArgumentException if the return type of {@code filter}
584      * is void, or it is not the same as the {@code pos} coordinate of the target var handle,
585      * if {@code pos} is not between 0 and the target var handle coordinate arity, inclusive,
586      * if the resulting var handle's type would have <a href="MethodHandle.html#maxarity">too many coordinates</a>,
587      * or if {@code filter} throws any checked exceptions.
588      */
589     public static VarHandle collectCoordinates(VarHandle target, int pos, MethodHandle filter) {
590         return JLI.collectCoordinates(target, pos, filter);
591     }

625         if (!carrier.isPrimitive() || carrier == void.class || carrier == boolean.class) {
626             throw new IllegalArgumentException("Illegal carrier: " + carrier.getSimpleName());

630     private static long carrierSize(Class<?> carrier) {
631         long bitsAlignment = Math.max(8, Wrapper.forPrimitiveType(carrier).bitWidth());
632         return Utils.bitsToBytesOrThrow(bitsAlignment, IllegalStateException::new);
633     }

636         if (!(carrier == byte.class || carrier == short.class || carrier == int.class)) {

638         }
639     }
