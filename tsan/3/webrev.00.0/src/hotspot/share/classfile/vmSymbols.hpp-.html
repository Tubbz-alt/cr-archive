 663 // Here are all the intrinsics known to the runtime and the CI.
 664 // Each intrinsic consists of a public enum name (like _hashCode),
 665 // followed by a specification of its klass, name, and signature:
 666 //    template(<id>,  <klass>,  <name>, <sig>, <FCODE>)
 667 //
 668 // If you add an intrinsic here, you must also define its name
 669 // and signature as members of the VM symbols.  The VM symbols for
 670 // the intrinsic name and signature may be defined above.
 671 //
 672 // Because the VM_SYMBOLS_DO macro makes reference to VM_INTRINSICS_DO,
 673 // you can also define an intrinsic's name and/or signature locally to the
 674 // intrinsic, if this makes sense.  (It often does make sense.)
 675 //
 676 // For example:
 677 //    do_intrinsic(_foo,  java_lang_Object,  foo_name, foo_signature, F_xx)
 678 //     do_name(     foo_name, "foo")
 679 //     do_signature(foo_signature, "()F")
 680 // klass      = vmSymbols::java_lang_Object()
 681 // name       = vmSymbols::foo_name()
 682 // signature  = vmSymbols::foo_signature()
 683 //
 684 // The name and/or signature might be a "well known" symbol
 685 // like "equal" or "()I", in which case there will be no local
 686 // re-definition of the symbol.
 687 //
 688 // The do_class, do_name, and do_signature calls are all used for the
 689 // same purpose:  Define yet another VM symbol.  They could all be merged
 690 // into a common 'do_symbol' call, but it seems useful to record our
 691 // intentions here about kinds of symbols (class vs. name vs. signature).
 692 //
 693 // The F_xx is one of the Flags enum; see below.
 694 //
 695 // for Emacs: (let ((c-backslash-column 120) (c-backslash-max-column 120)) (c-backslash-region (point) (point-max) nil t))
 696 //
 697 //
 698 // There are two types of intrinsic methods: (1) Library intrinsics and (2) bytecode intrinsics.
 699 //
 700 // (1) A library intrinsic method may be replaced with hand-crafted assembly code,
 701 // with hand-crafted compiler IR, or with a combination of the two. The semantics
 702 // of the replacement code may differ from the semantics of the replaced code.
 703 //
 704 // (2) Bytecode intrinsic methods are not replaced by special code, but they are
 705 // treated in some other special way by the compiler. For example, the compiler
 706 // may delay inlining for some String-related intrinsic methods (e.g., some methods
 707 // defined in the StringBuilder and StringBuffer classes, see
 708 // Compile::should_delay_string_inlining() for more details).
 709 //
 710 // Due to the difference between the semantics of an intrinsic method as defined
 711 // in the (Java) source code and the semantics of the method as defined
 712 // by the code in the VM, intrinsic methods must be explicitly marked.
 713 //
 714 // Intrinsic methods are marked by the jdk.internal.HotSpotIntrinsicCandidate
 715 // annotation. If CheckIntrinsics is enabled, the VM performs the following
 716 // checks when a class C is loaded: (1) all intrinsics defined by the VM for
 717 // class C are present in the loaded class file and are marked;
 718 // (2) an intrinsic is defined by the VM for all marked methods of class C;
 719 // (3) check for orphan methods in class C (i.e., methods for which the VM
 720 // declares an intrinsic but that are not declared for the loaded class C.
 721 // Check (3) is available only in debug builds.
 722 //
 723 // If a mismatch is detected for a method, the VM behaves differently depending
 724 // on the type of build. A fastdebug build exits and reports an error on a mismatch.
 725 // A product build will not replace an unmarked library intrinsic method with
 726 // hand-crafted code, that is, unmarked library intrinsics are treated as ordinary
 727 // methods in a product build. The special treatment of a bytecode intrinsic method
 728 // persists even if the method not marked.
