24 /**
25  * Send your graphs to <b>IGV</b> via a socket or a file. This package allows one to easily encode
26  * any graph-like data structure and send it for visualization to <em>OracleLab's Ideal Graph
27  * Visualizer</em> tool. Assuming you already have your own data structure that contains
28  * <b>nodes</b> and <b>edges</b> among them, creating a {@link org.graalvm.graphio.GraphOutput}
29  * specialized for your data is a matter of implementing a single interface:
30  *
31  * {@link org.graalvm.graphio.GraphJavadocSnippets#acmeGraphStructure}
32  *
33  * The {@link org.graalvm.graphio.GraphStructure} interface defines the set of operations that are
34  * needed by the <em>graph protocol</em> to encode a graph into the <b>IGV</b> expected format. The
35  * graph structure is implemented as a so called
36  * <a href="http://wiki.apidesign.org/wiki/Singletonizer">singletonizer</a> API pattern: there is no
37  * need to change your data structures or implement some special interfaces - everything needed is
38  * provided by implementing the {@link org.graalvm.graphio.GraphStructure} operations.
39  * <p>
40  * The next step is to turn this graph structure into an instance of
41  * {@link org.graalvm.graphio.GraphOutput}. To do so use the associated
42  * {@link org.graalvm.graphio.GraphOutput.Builder builder} just like shown in the following method:
43  *
44  * {@link org.graalvm.graphio.GraphJavadocSnippets#buildOutput}
45  *
46  * Now you are ready to dump your graph into <b>IGV</b>. Where to obtain the right channel? One
47  * option is to create a {@link java.nio.channels.FileChannel} and dump the data into a file
48  * (preferrably with <code>.bgv</code> extension). The other is to open a socket to port
49  * <code>4445</code> (the default port <b>IGV</b> listens to) and dump the data there. Here is an
50  * example:
51  *
52  * {@link org.graalvm.graphio.GraphJavadocSnippets#dump}
53  *
54  * Call the {@code dump} method with pointer to file {@code diamond.bgv} and then you can open the
55  * file in <b>IGV</b>. The result will look like this:
56  * <p>
57  * <img src="doc-files/diamond.png">
58  * <p>
59  * You can verify the behavior directly in the <b>IGV</b> by downloading
60  * <a href="doc-files/diamond.bgv">diamond.bgv</a> file generated from the above diamond structure
61  * graph.
62  * <p>
63  * The primary <b>IGV</b> focus is on graphs used by the compiler. As such they aren't plain graphs,
64  * but contain various compiler oriented attributes:
65  * <ul>
66  * <li>{@linkplain org.graalvm.graphio.GraphBlocks code blocks} information</li>
67  * <li>{@linkplain org.graalvm.graphio.GraphElements method and fields} information</li>
68  * <li>Advanced support for {@linkplain org.graalvm.graphio.GraphTypes recognizing types}</li>
69  * </ul>
70  * all these additional interfaces ({@link org.graalvm.graphio.GraphBlocks},
71  * {@link org.graalvm.graphio.GraphElements} and {@link org.graalvm.graphio.GraphTypes}) are
72  * optional - they don't have to be provided. As such they can be specified via
73  * {@link org.graalvm.graphio.GraphOutput.Builder} instance methods, which may, but need not be
74  * called at all. Here is an example:
75  *
76  * {@link org.graalvm.graphio.GraphJavadocSnippets#buildAll}
77  *
78  * All these interfaces follow the
79  * <a href="http://wiki.apidesign.org/wiki/Singletonizer">singletonizer</a> API pattern again - e.g.
80  * no need to change your existing data structures, just implement the operations provided by the
81  * interfaces you pass into the builder. By combining these interfaces together you can get as rich,
82  * colorful, source linked graphs as the compiler produces to describe its optimizations.
83  */
