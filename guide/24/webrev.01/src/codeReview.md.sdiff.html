 39 When you are asking reviewers to check over your proposed
 40 changes, it helps to make that job as easy for them as possible,
 41 and webrev is one way to do it.
 42 
 43 For more information about webrev, refer to:
 44 
 45 * See the [the webrev help page](webrevHelp.html)
 46 
 47 ## Example
 48 
 49 * A developer called "username" runs `webrev -f`
 50   on their work done in a Mercurial forest. This creates output under
 51   a webrev directory.
 52 
 53 * For clarity, they add the bug ID they are working on, and
 54   perhaps add a sequence number:
 55 
 56 >>     mkdir 6543210
 57 >>     mv webrev 6543210/webrev.00
 58 
 59   This will allow subsequent webrevs (if there are any) to be
 60   added as webrev.01 and so on without clobbering the original
 61   files.
 62 
 63 * **Optional:** If they would like to dress up their entry in
 64   the [Atom feed](https://cr.openjdk.java.net/feed.atom),
 65   create `.title` and/or `.description` files:
 66 
 67 >>     echo "A suitable review title or bug synopsis" > 6543210/.title
 68 >>     echo "Short description of the change to be reviewed" > 6543210/.description
 69 
 70   Syndication does not recognize line endings or formatting tags,
 71   so it would be a good idea to keep the information in these files
 72   as simple as possible.
 73 
 74   If these file(s) are not found by the cr.openjdk.java.net
 75   server, the subdirectory
 76   name (6543210 in this example) will be used for title and/or
 77   description in the stream.
 78 
 79 * They transfer the webrev tree to their home directory on
 80   cr.openjdk.java.net using scp or rsync:
 81 
 82 >>     scp -r 6543210 username@cr.openjdk.java.net:
 83 
 84 > or
 85 
 86 >>     rsync -av 6543210 username@cr.openjdk.java.net:
 87 
 88   Note the final : on the command line. If you omit that, you will
 89   copy the bits into a directory called
 90   "username@cr.openjdk.java.net" on your local system.
 91 
 92 * The results will be published at:
 93 
 94 >>     https://cr.openjdk.java.net/~username/6543210
 95 
 96 * In ten minutes or less, the new information will appear on the
 97   feed:
 98 
 99   [https://cr.openjdk.java.net/feed.atom](https://cr.openjdk.java.net/feed.atom)
100 
101 * Later on, if it is time to clean up, use sftp to access your
102   code review directory:
103 
104 >>     sftp username@cr.openjdk.java.net
105 
106   Use the `rm` command to delete individual files. If you
107   want to delete a tree of files and directories, use the
108   `rename` command to move them into your ~/.trash
109   subdirectory. To continue with the example for bug ID 6543210:
110 
111 >>     rename 6543210 .trash/6543210
112 
113   A cron job on the cr.openjdk.java.net
114   server will periodically empty the trash for
115   all users.
116 
117 Please send feedback to `web-discuss at openjdk dot java dot net`
