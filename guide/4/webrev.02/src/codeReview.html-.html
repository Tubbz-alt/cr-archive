 13     <h1>Community Code Review</h1>
 14 
 15     <h2>General</h2>
 16 
 17     <p>
 18       The cr.openjdk.java.net server provides storage and display of code review
 19       materials such as webrevs and other artifacts related to the
 20       <a href="http://openjdk.java.net/">OpenJDK</a> Community. If you
 21       are interested in monitoring recent reviews, try our review feed
 22       <a href="http://cr.openjdk.java.net/feed.atom">here</a>.
 23     </p>
 24 
 25     <p>
 26       Any user with push access to the OpenJDK
 27       <a href="http://hg.openjdk.java.net">Mercurial</a> server can publish
 28       materials on the cr.openjdk.java.net server.
 29       Users can upload files to temporary
 30       storage using secure methods (rsync, scp, and sftp).
 31     </p>
 32 
 33     <h2>Terms and conditions</h2>
 34 
 35     <p>
 36       This site is for open source materials related to the OpenJDK
 37       Community only. Users uploading inappropriate materials will lose
 38       access and the material will be deleted. Please review the <a href="http://openjdk.java.net/legal/terms.html">Terms of Use</a>.
 39     </p>
 40 
 41     <h2>Webrev</h2>
 42 
 43     <p>
 44       So, what is webrev?
 45     </p>
 46     
 47     <p>
 48       <a href="webrevHelp.html">Webrev</a> is
 49       a script that will crawl over your changes, compare the modified
 50       files with your parent forest, then generate a set of web pages to
 51       display the differences.
 52     </p>
 53     
 54     <p>
 55       When you are asking reviewers to check over your proposed
 56       changes, it helps to make that job as easy for them as possible,
 57       and webrev is one way to do it.
 58     </p>
 59     
 60     <p>
 61       For more information about webrev, refer to:
 62     </p>
 63     <ul>
 64       <li>
 65         See the <a href="webrevHelp.html">the webrev help page</a>
 66       </li>
 67     </ul>
 68 
 69     <h2>Example</h2>
 70 
 71     <ul>
 72       <li>
 73         <p>
 74 	  A developer called "username" runs <tt>webrev -f</tt>
 75           on their work done in a Mercurial forest. This creates output under
 76           a webrev directory.
 77 	</p>
 78       </li>
 79       <li>
 80         <p>
 81 	  For clarity, they add the bug ID they are working on, and
 82           perhaps add a sequence number:
 83 	</p>
 84         <pre>
 85           mkdir 6543210
 86           mv webrev 6543210/webrev.00
 87         </pre>
 88         <p>
 89 	  This will allow subsequent webrevs (if there are any) to be
 90           added as webrev.01 and so on without clobbering the original
 91           files.
 92 	</p>
 93       </li>
 94       <li>
 95         <p>
 96 	  <b>Optional:</b> If they would like to dress up their entry in
 97           the <a href="http://cr.openjdk.java.net/feed.atom">Atom feed,</a>
 98           create <tt>.title</tt> and/or <tt>.description</tt> files:
 99 	</p>
100         <pre>
101           echo "A suitable review title or bug synopsis" &gt; 6543210/.title
102           echo "Short description of the change to be reviewed" &gt; 6543210/.description
103         </pre>
104         <p>
105 	  Syndication does not recognize line endings or formatting tags,
106           so it would be a good idea to keep the information in these files
107           as simple as possible.
108 	</p>
109         <p>
110 	  If these file(s) are not found by the cr.openjdk.java.net
111 	  server, the subdirectory
112           name (6543210 in this example) will be used for title and/or
113           description in the stream.
114 	</p>
115       </li>
116       <li>
117         <p>
118 	  They transfer the webrev tree to their home directory on
119           cr.openjdk.java.net using scp or rsync:
120 	</p>
121         <pre>
122           scp -r 6543210 username@cr.openjdk.java.net:
123           		or
124           rsync -av 6543210 username@cr.openjdk.java.net:
125         </pre>
126         <p>
127 	  Note the final : on the command line. If you omit that, you will
128           copy the bits into a directory called
129           "username@cr.openjdk.java.net" on your local system.
130 	</p>
131       </li>
132       <li>
133         <p>
134 	  The results will be published at:
135 	</p>
136         <pre>
137             http://cr.openjdk.java.net/~username/6543210
138         </pre></li>
139       <li>
140         <p>
141 	  In ten minutes or less, the new information will appear on the
142           feed:
143 	</p>
144         <p>
145 	  <a href="http://cr.openjdk.java.net/feed.atom">http://cr.openjdk.java.net/feed.atom</a>
146 	</p>
147       </li>
148       <li>
149         <p>
150 	  Later on, if it is time to clean up, use sftp to access your
151           code review directory:
152 	</p>
153         <pre>
154           sftp username@cr.openjdk.java.net
155         </pre>
156         <p>
157 	  Use the <tt>rm</tt> command to delete individual files. If you
158           want to delete a tree of files and directories, use the
159           <tt>rename</tt> command to move them into your ~/.trash
160           subdirectory. To continue with the example for bug ID 6543210:
161 	</p>
162         <pre>
163           rename 6543210 .trash/6543210
164         </pre>
165         <p>
166 	  A cron job on the cr.openjdk.java.net
167 	  server will periodically empty the trash for
168         all users.
169 	</p>
170       </li>
171     </ul>
